## 一、核心设计原则

以下 6 条原则贯穿所有功能设计，同等重要，共同约束每一个产品决策。

### 关系优先于活动

串门是一个帮人交朋友的社群，不是活动平台。活动是手段，关系是目的。产品设计始终把"人与人之间的关系"放在活动之上——让关系可见、可追溯、可加温。

### 个性化引导降低参与成本

不是等人自己来，而是在对的时机推一把。通过 rule-based Email 通知系统，在对的时机用朋友的语气推送引导，而不是群发通知。

### 轻输入，重整理

用户的输入永远是简单的（发一条动态、贴个链接、点个投票）。平台负责把这些输入整理成结构化的内容。用户觉得自己在"发朋友圈"，系统知道这是一条有类型、有关联、可参与的内容。

### 让参与可见

每个人的贡献——推荐、投票、Host、报名、支持——都被系统记录和呈现。不只是"发了就沉了"，而是被整理成清晰的画面：你推荐了多少部电影、参加了多少场活动、有多少人投了你推荐的片子。

### 每条内容都是参与入口

动态流里的每条内容都带有可操作的动作。看到电影推荐 → 投票。看到活动 → 报名。看到提案 → 响应。不是被动浏览，是随时参与。

### 共创而非分发

所有面向成员的内容来自成员自身的贡献，不是运营单方面的输出。平台的价值来自所有成员的参与叠加，而非中心化的内容生产。

## 二、产品定位

### 一句话

串门网页端是一个让社群成员共同创造内容、共同决定做什么、共同记录经历的平台。

### 终极形态

一个基于共同价值观的私域社交平台。前台是轻松的动态流，人人随手参与；后台是结构化的社群知识库，把所有人的参与整理成清晰的集体成果。每个人都能看到自己在集体中的位置。

### V1 要验证的核心假设

> **"让参与可见"这件事，真的能让人更想参与吗？**

### 与现有工具的关系

- **微信群**：继续承担日常聊天，网页端不替代即时通讯
- **Notion**：历史数据来源，一次性迁移后退役为运营内容编辑工具（串门原则、Host 指南等）
- **小红书/Luma**：继续承担对外拉新，网页端服务已在社群内的人

## 三、两个核心循环

串门网页端的功能围绕两个循环展开，它们在"活动"节点交汇，在"人"的维度沉淀。

### 循环1：推荐 → 响应 → 选中 → 完成

一个人的品味或想法，经过集体的响应和筛选，变成所有人共享的体验。

```
某人推荐了一部电影
  → 其他人投票"我也想看"
    → Host 办电影夜时选中了它
      → 大家一起看了，变成共同记忆
```

**V1 载体：电影推荐池、活动提案**

### 循环2：发起 → 报名 → 参与 → 记录

一个人愿意打开门，一群人走进来，一段经历被记录下来。

```
某人发起一场活动
  → 成员报名
    → 线下一起做了
      → 照片、参与者、内容被记录
```

**V1 载体：活动日历、活动记录**

### 交汇与沉淀

两个循环在"活动"节点交汇——循环1 产出"我们想做什么"，循环2 产出"我们做了什么"。而每次交汇都沉淀出人与人之间的关系数据：共同品味、共同经历、互相感谢。这些关系数据通过"人与人连接层"（见 5.5）反哺下一次循环。

### 4.1 首页（动态流）

**定位**：社群的心跳。聚合所有模块的更新，让人每天想来看看。

**内容来源**（从各模块聚合）：

- 🎬 有人推荐了一部新电影
- 👍 一部电影获得了新投票
- 💡 有人提了一个新活动提案
- 📅 有人发起了一场新活动
- 📸 一场活动刚结束，有了回顾/照片
- ✉ 有人寄了一张公开感谢卡
- 🌱 ~~有人的种子计划有了新进展~~（V2）
- 💬 有人发起了一个话题讨论
- 🎲 本周小局抽签结果 / 小局完成记录
- 🎉 集体里程碑（见 5.5 集体里程碑）

**视觉层级**：

动态流不是所有卡片等高等宽。三层视觉权重让用户在滚动时有节奏感：

| 层级 | 适用内容 | 视觉特征 |
|------|---------|---------|
| Tier 1 · 高亮 | 系统横幅、集体里程碑 | 全宽、边框发光、最醒目 |
| Tier 2 · 标准 | 活动、活动回顾、感谢卡、讨论 | Card 组件包裹，标准内边距 |
| Tier 3 · 紧凑 | 电影推荐、活动提案 | 无 Card 包裹，s1 底色 + 细边框，更矮。同类内容聚合为一组（如"🎬 新片推荐"） |

**时间分隔线**：动态按时间倒序，用"今天"、"本周"、"更早"分隔线划分区段。分隔线样式为左右细线 + 居中文字，视觉极轻。

**点击穿透**：Feed 中的活动卡片、电影推荐、提案均可点击跳转到对应详情页面（活动→活动页，电影→推荐页，提案→活动页提案 tab）。投票/报名等按钮有 stopPropagation，不会误触跳转。

**头像穿透**：全站所有页面中出现的头像和用户名均可点击，跳转到该成员的详情页（与我的共同经历）。这是进入他人 profile 的主要入口，不需要绕道成员墙。动态流、活动详情、感谢卡、讨论回复中的头像和名字均适用。

**空状态（新成员首次登录）**：

新成员通过审核后首次登录（Google 或 magic link），看到的不是空白页面，而是一个欢迎引导：
- 系统欢迎横幅："欢迎来串门儿，{display_name} 👋" + 介绍串门儿是什么
- 四个快捷入口：看看活动 / 推荐电影 / 了解串门 / 寄感谢卡
- 底部提示："动态流会随着你参加活动慢慢丰富起来 ✨"

**每条动态的结构**：

```
[头像] [用户名] [动作描述] [时间]
[内容卡片（根据类型不同呈现不同样式）]
[参与动作按钮] [点赞] [评论]
```

**参与动作（因内容类型而异）**：

| 内容类型 | 参与动作 |
|---------|---------|
| 电影推荐 | "我也想看"投票 |
| 活动提案 | "感兴趣"投票 |
| 活动发起 | 报名 / Waitlist |
| 活动回顾 | 点赞、评论 |
| 公开感谢卡 | 点赞 |
| 话题讨论 | 回复、点赞 |

**通用动作**：点赞、评论（所有内容类型都支持）

**排序逻辑**：按时间倒序，V1 不做算法推荐。运营手动或规则自动生成的社群级内容（集体里程碑、Host 致敬、供给求助）可置顶。

**发起话题入口**：动态流顶部固定一个轻量输入栏（placeholder："聊点什么..."），点击展开话题发起表单（标题 + 正文）。提交后即作为话题讨论卡片出现在动态流中。

**V1 数据来源**：从各数据表聚合查询生成动态卡片（见第六章）

### 4.2 推荐池（电影 + 多品类）

**定位**：社群共同积累的品味库。位于「推荐」页面，支持 6 个品类。

**品类选择器**：推荐页顶部显示横滑品类选择器，6 个品类全部可用：🎬 电影 / 📖 书 / 🍽️ 食谱 / 📍 地方 / 🎵 音乐 / 🎭 演出和其他。电影品类使用独立的 Movie 数据模型和 TMDB 搜索；其他品类使用统一的 Recommendation 数据模型和 RecommendationVote 投票。所有品类支持浏览、搜索、投票、推荐、编辑、删除。推荐可被 Host 关联到活动（见 EventRecommendation）。

**浏览视图**：

- 所有电影列表，显示：封面、片名、年份、豆瓣评分、推荐人、投票数、状态（候选/已放映）
- 排序：默认按投票数降序，可切换为最新推荐、最近放映
- 筛选：候选 / 已放映 / 全部

**用户动作**：

| 动作 | 说明 | 需要登录 |
|------|------|---------|
| 浏览 | 查看所有电影及详情 | 否 |
| 投票"我想看" | 对任意候选电影投票，投票可取消 | 是 |
| 推荐新电影 | 贴豆瓣链接，系统自动抓取信息（片名、封面、导演、年份、评分），加入池子 | 是 |

**电影信息抓取容错**：
- 抓取成功：自动填充所有字段，用户确认后提交
- 抓取失败（豆瓣反爬、链接格式错误、网络超时）：提示"自动获取失败，请手动填写"，展开手动填写表单
- 手动填写必填字段：片名、年份、导演。可选：类型、时长、海报图片上传、评分
- 支持 TMDB 搜索（自动填充片名、海报、导演、年份等）+ 手动填写
| 提名到活动 | 把电影推荐到某次电影夜（关联活动） | 是 |

**电影详情页**（点击电影列表中的任意电影进入）：

- **大海报 Hero 区域**：使用电影渐变海报（`posters[title]`）作为全宽背景，高度 280px，底部叠加渐变遮罩，在遮罩上显示片名（海报 accent 色）、年份、导演、英文副标题。Hero 下方紧接 Chips 栏：类型 / 时长 / 评分 / 放映状态标签
- **简介**：电影概要文字
- **推荐人**：头像 + 名字，整行可点击跳转至成员详情页（`/members/{name}`）
- **投票区域**：投票按钮 + 总票数 + 投票人头像列表（`AvatarGroup`），每个头像可点击跳转至成员详情页
- **放映关联**（自动连接逻辑）：从所有活动数据源（`upcomingEvents` / `liveEvents` / `endedEvents` / `pastEvents`）中通过 `film` 字段匹配当前电影，分三组显示：
  - **即将放映**：来自 `upcomingEvents` 中 phase 为 open / invite / closed 的匹配活动
  - **正在进行**：来自 `liveEvents` 中的匹配活动，带绿色圆点指示器
  - **历史放映**：合并 `endedEvents` + `pastEvents` + `MovieDetailData.screenings`，去重后展示
  - 每个活动卡片可点击跳转至活动详情页
- 未放映过的电影显示空状态："还没有放映过" + "🎬 提名到下次电影夜"按钮
- **讨论区**：评论按时间顺序展示，每条评论可带来源标签（如"来自「电影夜 · 寄生虫」→"，跳转到对应活动详情页）。底部有评论输入框

**Host 选片视图**（在活动页面内，详见上方"活动内电影提名与投票"）：

对每部被提名的电影显示：
- 总投票数（社群整体偏好）：moviePool.v
- 本场报名者投票数（参加这次活动的人中多少人想看）：`movie.voters.filter(name => event.people.includes(name)).length`

**状态流转**：

```
候选（pool）→ 已放映（screened）
```

**触发机制**：
- Host 在活动详情页"选片"操作中选定一部电影 → 写入 `movie_screenings` 表（movie_id + event_id）
- 活动 phase 变为 'ended' 时 → cron 自动将该电影 status 从 `pool` 改为 `screened`
- 一部电影可以被多次放映（screened 后不阻止再次被选）

**"提名到活动"交互**：

```
用户在电影详情页点击"🎬 提名到下次电影夜"
  → 弹出选择面板：显示当前 phase='invite' 或 'open' 且 tags 包含"电影"的活动列表
  → 如果有可选活动：选择一场 → 写入 movie_nominations（status='nominated'）→ Host 在选片视图中看到提名
  → 如果没有电影夜活动：提示"暂时没有计划中的电影夜，你可以发起一个提案 →"
  → 每部电影每场活动仅可被提名一次
```

### 4.3 活动提案

**定位**：任何人都能提出想法，社群投票响应，有人认领执行。位于「活动」页面的「提案」子页签。

**统一创建入口**：活动页底部有一个"+"浮动按钮（SpeedDial FAB），点击展开两个选项：
- **📅 发起活动**：直接创建正式活动（→ `/events/new`，EventRoundedIcon）
- **💡 提一个想法**：发起提案，等待社群响应（→ `/events/proposals/new`）

**动态流通用创建入口**：首页（FeedPage）同样提供固定位置 SpeedDial（仅登录用户可见），包含四个快捷动作：
- **📅 发起活动** → `/events/new`
- **💡 活动创意** → `/events/proposals/new`
- **✉ 寄感谢卡** → `/cards`
- **👍 推荐电影** → `/discover/movie/add`

**浏览视图**：

- 所有提案列表，显示：标题、提出者、投票数、状态（讨论中/已排期/已完成/取消）
- 排序：默认讨论中的在前，按投票数降序

**用户动作**：

| 动作 | 说明 | 需要登录 |
|------|------|---------|
| 浏览 | 查看所有提案 | 否 |
| 投票"感兴趣" | 对任意讨论中的提案投票 | 是 |
| 提交新提案 | 写标题 + 描述，发起一个新想法 | 是 |
| 🏠 我来组织 | 认领一个提案，转为正式活动 | 是 |
| 评论 | 在提案下方讨论 | 是 |

**想法→活动转化**：提案卡片上同时显示"🙋 感兴趣"和"🏠 我来组织"两个按钮。任何人点击"我来组织"后，系统创建一个预填了提案信息的活动草稿，组织者补充时间、地点、人数后发布。原提案状态自动流转为"已排期"。

**状态流转**：

```
讨论中 → 已排期（有人认领，创建了对应活动）
讨论中 → 已完成
讨论中 → 取消
```

**与活动的关联**：提案被认领后，创建对应的活动日历条目，两者关联。

### 4.4 活动日历

**定位**：接下来要发生什么，以及报名参加。位于「活动」页面的「即将到来」子页签。

**关键约束**：活动建议人数上限不超过 10 人。小规模是串门的核心体验，保持亲密感。

#### 本周小局（抽签做东机制）

活动页「即将到来」顶部固定一张"本周小局"卡片，展示本周抽签结果。

**机制**：
- 系统每周从活跃成员中随机抽签一人，作为本周"做东人"
- 做东人收到通知："轮到你了！"
- 约谁、做什么、几个人，完全由做东人决定
- 做东人点击"发起小局"→ 进入轻量版活动创建流程（预填 Tag 为"小局"）
- 见面后简单记录 → 出现在 feed

**解决的问题**：给"谁都不好意思先开口"一个正当理由——不是我主动约你，是我被抽中了必须约人。

**卡片展示**：
- 🎲 图标 + "本周小局" 标题 + 期数
- 做东人头像和名字
- "轮到你做东了！"提示
- 三个按钮：发起小局 / 这周不行（跳过后系统重新抽签）/ 历史记录

**数据**：抽签记录（期数、被抽中人、是否完成、关联活动 ID）

**活动页四状态切换**：

活动页顶部有四个 Tab 切换不同状态的活动列表：

| Tab | 内容 | 说明 |
|-----|------|------|
| 报名中 | 正在接受报名/邀请的活动 | 包含本周小局卡片（含"发起小局"+"这周不行"按钮） |
| 进行中 | 当前正在发生的活动 | 显示绿色 ● 标签、已进行时长 |
| 想法孵化中 | 活动提案列表 | 独立 tab 展示所有提案，含"添加想法"按钮、感兴趣投票、"我来组织"认领 |
| 已结束 | 历史活动回顾 | 紧凑列表，显示照片数/评论数 |

**Tab 与 events.phase 映射**：

| Tab | 对应 phase | 排序 |
|-----|-----------|------|
| 报名中 | `invite` + `open` | 按活动日期升序（最近的排前面） |
| 进行中 | `live` | 按开始时间升序 |
| 已结束 | `ended` | 按活动日期降序（最近结束的排前面） |

`cancelled` 状态的活动不出现在任何 Tab。管理后台可在活动管理中筛选查看已取消活动。

**活动卡片**（列表视图）：
- 活动名称、日期时间、Tags（可多选：电影夜/Potluck/徒步/咖啡/运动/小局/其他）、Host、剩余名额
- 场景头图（氛围渐变占位）
- 报名中：显示剩余名额 + 参与者头像
- 进行中：显示 ● 进行中标签 + 已进行时长
- 已结束：紧凑行布局，显示 📷 照片数 + 💬 评论数

**活动详情页**（三种状态共用一个详情页，根据状态显示不同内容）：

**通用区域（三种状态都显示）**：
- 场景头图 + 状态标签（🔒 Host 专属邀请 / ● 进行中 / 已结束）
- 活动基本信息（名称、日期、地点）

**电影提名与投票 — 后端数据需求**：

```
── 数据表 ──

movie_nominations {
  id:          PK
  event_id:    FK → events.id           -- 关联活动
  movie_id:    FK → movies.id           -- 被提名的电影
  nominated_by: FK → users.id           -- 提名人（必须是该活动 status='accepted' 的参与者）
  created_at:  timestamp
  UNIQUE(event_id, movie_id)            -- 每部电影每场活动仅可提名一次
}

movie_votes {
  id:          PK
  movie_id:    FK → movies.id           -- 被投票的电影
  user_id:     FK → users.id            -- 投票人
  created_at:  timestamp
  UNIQUE(movie_id, user_id)             -- 每人每部电影仅可投一次
}

── API 端点 ──

POST   /events/:eventId/nominations     -- 提名电影到活动（body: { movieId }）
DELETE /events/:eventId/nominations/:movieId  -- 取消提名
GET    /events/:eventId/nominations     -- 获取本场提名列表 + 投票数据

POST   /movies/:movieId/vote            -- 投票（活动页 & 电影详情页共享）
DELETE /movies/:movieId/vote            -- 取消投票

── 查询逻辑 ──

-- 获取某活动的候选电影 + 双维度投票
SELECT
  m.id, m.title, m.year, m.director,
  COUNT(DISTINCT mv.user_id) AS total_votes,           -- 社区总投票
  COUNT(DISTINCT CASE
    WHEN es.status = 'accepted' THEN mv.user_id
  END) AS attendee_votes                                 -- 本场参与者投票
FROM movie_nominations mn
JOIN movies m ON m.id = mn.movie_id
LEFT JOIN movie_votes mv ON mv.movie_id = m.id
LEFT JOIN event_signups es ON es.event_id = mn.event_id
  AND es.user_id = mv.user_id
WHERE mn.event_id = :eventId
GROUP BY m.id
ORDER BY attendee_votes DESC, total_votes DESC;

── 前端数据映射 ──

EventData.nominations: number[]         -- 提名的 movie_id 列表
MovieDetailData.voters: string[]         -- 所有投票人（用于 filter 计算本场票数）
本场投票数 = movie.voters.filter(name => event.people.includes(name)).length
```

#### 活动发起流程（两阶段邀请机制）

活动发布不是一次性广播，而是分阶段释放名额，让 Host 对参与者有控制感：

**第一阶段：Host 创建 + 定向邀请**

```
Host 创建活动
  → 系统引导：填写名称、日期、地点、人数上限（建议≤10）、Tag（可多选）、说明、House Rules
    → **地点字段**："在我家" toggle。开启后自动填入 Host 的 `home_address`，此地址仅对报名成功（status='accepted'）的参与者可见；其他人看到的地址显示为 Host 的 `location`（城市/地区）。关闭则手动输入活动地点（公开场所等）
    → **House Rules 自动填入**：如果 Host 个人资料中有 `default_house_rules`，创建时自动填入，Host 可修改
  → 系统引导："想先邀请谁？"
    → 系统提供建议（基于后端社交上下文数据）：
      "以下成员参加过你上次的活动且待到了最后：XX、YY、ZZ"
      "以下新成员还没参加过你的活动类型，可以考虑邀请"
  → Host 从成员列表选择想定向邀请的人
  → 被邀请人收到私人通知："XX 邀请你参加 XX 活动"
  → 被邀请人可接受 / 拒绝 / 暂不回复
```

此阶段活动仅对被邀请人可见。Host 可以先锁定自己最想邀请的核心参与者。

**Host 邀请管理视图**：

Host 在活动详情页的管理面板中可以实时查看所有邀请的响应状态：

```
管理 ▾ → 邀请管理

邀请状态列表：
  ✅ 已接受：[头像] 星星 · 2/20 接受
  ⏳ 未回复：[头像] 白开水 · 已邀请 2 天
  ❌ 已婉拒：[头像] 大橙子 · 2/19 婉拒

操作：
  - 对"未回复"的人：[再提醒一次]（7 天内仅可提醒 1 次）
  - [邀请更多人]（跳转成员选择列表）
  - [提前结束邀请，公开活动]（确认后 phase → 'open'）
```

**邀请提醒**：Host 点击"再提醒一次"后，被邀请人在下一封每日邮件中收到 P0-A 重发。每个被邀请人 7 天内最多被提醒 1 次。

**Host 可见性控制**：Host 在创建活动时可以设置"不对以下成员显示"名单。被排除的成员看不到这场活动的存在——不是"报名被拒"，而是根本不知道有这场活动。比"拒绝"更体面，保护双方面子。此设置仅 Host 和运营可见。

**第二阶段：公开广播（活动前约一周，具体时间待定）**

```
定向邀请窗口结束
  → 系统自动将活动发布到动态流 + 活动日历
  → 对所有可见成员公开
  → 剩余名额开放报名（先到先得）
  → 满员后自动切换为 Waitlist 模式
  → 公开阶段 Host 仍可审核报名，接受或拒绝
```

**参与前引导（首次报名触发）**

当用户第一次报名某场活动时，系统在确认报名前插入一个轻量引导页：

```
报名确认前展示：
  1. 串门原则摘要（3-5 条核心要点，不是全文）
     - 真诚是前提
     - 对 Host 礼貌（准时、遵守 house rules、帮忙收拾）
     - 不推销、不越界
  2. 本场活动的 house rules（Host 写的注意事项）
  3. "我已了解" → 确认报名
```

非首次参加的用户跳过原则摘要，只展示本场 house rules（如果 Host 写了的话）。

**Waitlist 机制**：

满员后用户可加入 Waitlist。Waitlist 同时是社群供给不足的信号——Email 系统会根据 Waitlist 情况触发 Host 挖掘引导（见 5.4 P3-D）。

**Waitlist 递补状态流转**：
```
有人取消报名 → Waitlist 第一位的 status 改为 'offered' + 写入 offered_at
  → 发送 TXN-3（"有位置了"）
  → 24h 内点击 [确认参加] → status 改为 'accepted'
  → 24h 未响应 → cron 自动改为 'declined' → 下一位递补
```

**报名状态枚举说明**：
| 状态 | 含义 | 触发 |
|------|------|------|
| invited | Host 定向邀请，等待响应 | Host 创建活动时选人 |
| accepted | 已报名/已接受邀请 | 用户报名或接受邀请 |
| waitlist | 排队中 | 满员后报名 |
| offered | Waitlist 递补，等待确认 | 有人取消后自动 |
| rejected | 被 Host 拒绝 | Host 审核报名 |
| declined | 婉拒邀请 / 递补超时 | 用户操作或 24h cron |
| cancelled | 用户主动取消报名 | 用户操作 |

**取消报名通知**：用户取消报名后，Host 在下一封每日邮件中收到提醒："{user} 取消了「{event_name}」的报名"（纳入 P0 层级，因为影响 Host 的人数安排）。如果该用户是 Host 定向邀请的人，提醒文案改为："{user} 婉拒了你的邀请"。

**Waitlist 递补确认落地页**：TXN-3 中的 [确认参加] 链接跳转至活动详情页，页面顶部显示高亮横幅："你有一个递补名额，24h 内确认 → [确认参加] [放弃]"。非首次参加者无需再阅读串门原则。确认后横幅消失，显示正常已报名状态。

**活动 Phase 生命周期**：

活动从创建到结束经历 5 个 phase，由 Host 手动驱动：

```
Host 创建活动（填写 start_time / end_time / invite_deadline）
  → phase = 'invite'（仅被邀请人可见）
  → Host 手动"公开报名" → phase = 'open'
  → Host 手动"关闭报名" → phase = 'closed'
  → Host 手动"结束活动" → phase = 'ended'
  → 任意阶段 Host/管理员可手动取消 → phase = 'cancelled'
```

**Phase 转换 cron**：EventBridge 每 15 分钟触发一次 Lambda，扫描需要状态转换的活动。

**创建活动时必填时间字段**：
- `start_time`：活动开始日期时间
- `end_time`：活动结束日期时间（默认 = start_time + 3h，可调整）
- `invite_deadline`：定向邀请截止（默认 = start_time - 7 天，可调整。设为"立即公开"则跳过邀请阶段直接 phase='open'）

**贡献角色指定**：

Host 在活动详情页（Host 视角）的管理面板中可指定贡献角色：
- **入口**：活动详情页 → "管理 ▾" 下拉菜单 → "贡献角色"

**"管理 ▾" 下拉菜单完整内容**（仅 Host/Co-Host/Admin 可见）：

| 菜单项 | 说明 | 可用 Phase |
|--------|------|-----------|
| 邀请管理 | 查看邀请响应状态，再提醒，提前公开 | invite |
| 报名管理 | 查看报名列表，接受/拒绝，管理 Waitlist | invite, open |
| 编辑活动 | 修改时间、地点、人数、描述、house rules | invite, open, live |
| 选片 | 从提名列表中选定放映电影（仅电影夜） | invite, open |
| 贡献角色 | 指定 Co-Host / 记录人 / 设备 / 其他 | 所有 phase |
| 发布求助 | 发"需要帮忙"（搬设备、带食物等） | invite, open, live |
| 不可见名单 | 管理哪些成员看不到这场活动 | invite, open |
| 取消活动 | 取消整场活动（二次确认 + 通知已报名者） | invite, open, live |
| 转让 Host | 把 Host 身份转给已报名的某人 | invite, open, live |
- **操作**：从该活动参与者列表中选人 → 指定角色（Co-Host / 📷 记录 / 🔧 设备 / 其他）
- **时机**：活动前、活动中、活动后均可操作
- Co-Host 指定后立即获得活动管理权限；其他角色仅用于展示和 Email 引导
- 数据写入 `event_roles` 表

**用户动作**：

| 动作 | 说明 | 需要登录 |

#### 小局（PopupHost 乱盘）✅ V1

**本质**：解决"谁都不好意思先开口约人"的问题。系统每周抽签指定一个人做东，给 TA 一个正当理由去主动约人。

**核心概念——"线"（Line）**：每条线是一个独立的连续抽签序列。系统可以同时运行多条线，每条线独立 track 已连续进行了多少期。

**核心流程**：

```
系统每周一 cron 抽签 → 每条线各抽一个人（潜在 Host）
  → 立即发送 TXN-6 邮件："🎲 轮到你做东了！约谁、做什么、几个人，你说了算。[发起小局 →] [这周不行 →]"
  → 活动页置顶卡片同步显示（多条线 = 多张卡片）
  → 被抽中的人点击 [发起小局] → 进入轻量版发起活动流程
  → 或点击 [这周不行] → 系统立即重抽下一位 → 新的被抽中者收到 TXN-6
  → 见面后简单记录 → 出现在动态流
```

**并发支持**：
- 支持同时运行多条小局线（例如：线 A 第 12 期、线 B 第 5 期）
- 每条线独立抽签、独立 track 连续期数
- 活动页「报名中」tab 为每条活跃线各显示一张置顶卡片
- 管理员可在后台创建/暂停/恢复线

**连续期数 tracking**：
- 每条线记录 `streak_count`（已连续进行的期数）
- 如果某一周所有候选人都跳过导致本周空缺，`streak_count` 归零，下周重新计数
- 置顶卡片上显示"每周抽签 · 第 N 期"，让社区看到这条线的持续性
- 连续期数是社群活跃度的信号——"我们已经连续 12 周有人约人了"

**通知渠道**：TXN-6（事务性邮件，立即发送）。不依赖用户主动打开网页。

**轻量版发起活动流程**：复用正式活动创建，以下字段预填/简化：
- Tag 预填"小局"
- 跳过定向邀请阶段（phase 直接为 'open'）
- 人数上限默认 4（可调）
- 无 house rules 字段（可选填）
- 其他字段同正式流程（名称、日期时间、地点、说明）

**跳过机制**：
- 被抽中者在 TXN-6 邮件或活动页置顶卡片中点击"这周不行"
- 系统立即标记 `lottery_draw.status = 'skipped'`，同一 cron 周期内为该线重抽
- 重抽遵循相同规则（最近 4 周不重复、从未 Host 优先）
- 连续 2 周跳过的人降低被抽中概率（权重减半）
- 若某条线所有候选人都跳过，该线本周空缺（streak 归零），不强制

**与普通活动的区别**：普通活动是"有人主动发起"，小局是"机制指定一个人，给 TA 发起的理由"。不是配对，不是强制，是降低开口门槛。

**页面呈现**：

活动页「报名中」tab 顶部为每条活跃线各置顶一张卡片——「🎲 本周小局」：
- 显示线名称 + 连续期数（如"每周抽签 · 第 12 期"）
- 显示被抽中的人（头像、名字）
- 文案：可由线的描述自定义，默认"轮到你做东了！约谁、做什么、几个人，你说了算。"
- 主按钮："🏠 发起小局"→ 进入轻量版发起活动流程（预填 Tag 为"小局"）
- 次按钮（仅被抽中者可见）："这周不行"→ 确认弹窗 → 系统为该线重抽
- 链接："查看历史"→ 查看该线的过往小局记录

**抽签规则**：
- 每周一 EventBridge 触发 Lambda 自动抽签，为每条活跃线各抽一人
- 同一个人在同一周内不会被多条线同时抽中
- **活跃成员定义**：过去 90 天内参加过 ≥1 场已结束活动（`event_signups.status = 'accepted'` + `events.phase = 'ended'`）的 approved 成员
- 每条线内最近 4 周已被抽中的人不重复
- 从未 Host 过的人优先（更高权重）——小局是零压力的 Host 入门体验

**数据模型**：
```
lottery_lines: { id, name, description, streak_count, status: 'active'|'paused', created_at }
lottery_draws: { id, line_id, week_number, user_id, status: 'pending'|'accepted'|'skipped'|'completed', created_at }
```

**与 Host 培养的关系**：小局是最轻量的 Host 体验。约两三个人喝杯咖啡，就完成了一次"做东"。Email 系统可以在小局结束后引导："上次小局玩得开心吗？要不要试试在家办一次电影夜？"——从小局自然过渡到正式 Host。

## 五、通用功能与核心系统

### 5.1 注册、登录与身份

#### 注册方式：申请制

串门儿采用申请制。任何人可以提交申请，管理员审核通过后才能成为成员。这是"对的人 > 更多人"原则的第一道门。

**申请入口**：
- 网页端：关于串门儿页面底部 + 每个页面的 header 右上角（未登录时显示"申请加入"按钮）
- 外部渠道：小红书、串门来信等放置申请链接

**申请流程**：

```
访客点击"申请加入"
  → 填写申请表（见下方字段）
  → 提交后看到："已收到，我们会在 3 天内通过 Email 回复你 ✉"
  → 管理员在后台审核
  → 进入公示 → status = 'announced'，动态流显示"👋 欢迎新朋友"介绍卡片
    → 3 天公示期结束 → cron 自动批准 → status = 'approved'
    → 管理员可手动提前批准
    → 动态流显示"🎉 正式成为串门人"欢迎卡片
  → 直接通过 → 系统发送欢迎邮件（TXN-4），含网站链接，用 Google 账号或 magic link 登录
  → 拒绝 → 系统发送婉拒邮件（TXN-5，语气温和，不解释原因）
  → 7 天未处理 → 管理员后台标黄提醒
```

**申请表字段**（从现有 Tally 表单迁移）：

| 字段 | 类型 | 必填 | 映射到 users 表 | 成员墙展示 | 说明 |
|------|------|------|----------------|-----------|------|
| 怎么称呼你？ | 文本 | ✅ | `display_name` | ✅ | 显示名，不要求实名 |
| 你在哪个城市或地区？ | 文本 | ✅ | `location` | ✅ | 如"Edison, NJ" |
| 和大家打个招呼吧 | 长文本 | ✅ | `bio` | ✅ | 自我介绍 |
| 你觉得自己是一个什么样的朋友？ | 长文本 | ✅ | `self_as_friend` | ✅ | 审核参考 + 个人页面 |
| 你最好的朋友是什么样子的？ | 长文本 | ✅ | `ideal_friend` | ✅ | 审核参考 + 个人页面 |
| 你可能会怎么参与串门儿 | 长文本 | ✅ | `participation_plan` | ✅ | 审核参考 + 个人页面 |
| 封面图 | 图片上传 | ❌ | `cover_image_url` | ✅ | 个人页面封面，仅成员可见 |
| 从哪知道串门儿的？ | 文本 | ❌ | `referral_source` | ❌ | 运营分析用，仅后台可见 |
| 微信号 | 文本 | ✅ | `wechat_id` | ❌ | 运营联络用，仅后台可见 |
| Email | 文本 | ✅ | `email` | ✅ | 登录凭证 + 成员墙展示（用户可在设置中关闭） |

**前端展示规则**：
- 成员墙 / 成员详情页展示除微信号、referral_source 和 Email 以外的所有字段
- Email 默认在成员墙/成员详情页对其他已登录成员展示，用户可在账号设置中关闭（`hide_email` toggle）
- 微信号仅管理后台可见，前端不展示
- `user_status`（审核状态）仅管理后台可见，前端不展示任何审核相关信息
- 已登录成员可在"我的页面"编辑自己的展示字段（display_name、location、bio、self_as_friend、ideal_friend、participation_plan、cover_image、email）

**审核标准（给管理员的内部参考，不对外展示）**：
- 自我介绍是否认真写了（不是敷衍一句话）
- "什么样的朋友"和"怎么参与"是否体现真诚意愿
- 不做硬性门槛筛选（不按职业、学历、地域过滤）
- 有疑虑时优先通过，让实际参与来检验

**重复申请**：被拒绝的申请者 30 天后可重新申请。被封禁的成员不可申请。

**申请表页面设计**：

- **入口**：Header 右上角"申请加入"按钮（未登录时） / 关于串门儿页面底部 CTA / 外部链接
- **页面形态**：独立页面（`/apply`），单页长表单，不分步
- **页面结构**：
  - 顶部引导区（申请表和「关于串门儿」页面共用此文案）：

    > 你好，欢迎来串门儿！
    >
    > 串门儿是一个重建真实联结的计划，是个申请制的共创社区，欢迎你的加入！
    >
    > - 相比对这个世界的不满，我们更希望通过自己做一些事，让这个世界更接近我们想要的样子
    > - 相比萍水相逢，我们更希望能够有深度的交流和长久的联系
    > - 相比一起吃喝玩乐，我们更希望能真诚的在乎彼此，提供力所能及的帮助
    > - 相比其乐融融，我们更希望礼貌但真诚的表达和沟通
    >
    > 感谢你愿意花时间回答问题，问卷大概需要15分钟时间完成。你的回答我们会认真阅读，并给予回应。
    >
    > 如果通过申请，我们会邀请你进入"共创群"，这份问卷的内容会分享给所有会员，同时你也会看到其他朋友的答案。

  - 表单区：按申请表字段顺序排列，每个字段有标签和提示文案
  - Email 字段下方提示："建议填写 Google 邮箱，方便后续一键登录"
  - 封面图上传：拖拽或点击上传，预览缩略图，非必填标注"可选"
  - 底部：提交按钮 + "已有账号？去登录"链接
- **提交后**：全屏确认页——"✉ 已收到你的申请" + "我们会在 3 天内通过 Email 回复你" + "在此期间，可以先看看串门儿是什么 →"（跳转关于串门儿）
- **表单验证**：必填字段为空时提示，Email 格式校验，长文本字段无字数限制但建议 ≥20 字

#### 登录方式：Google 为主，Magic Link 备选

- **Google 登录**（主要）：一键登录，系统通过 `google_id` 或 Google 账号的 email 匹配 users 表记录
- **Email Magic Link**（备选）：输入注册 email → 收到 magic link → 点击登录。有效期 15 分钟，一次性使用
- 登录后 session 保持 30 天（可配置）
- 未通过审核的 email 登录后提示"你的申请正在审核中，我们会在 3 天内通过 Email 回复你"
- 未申请的 email 登录后提示"还没有账号？去申请加入 →"

**账号关联机制**：

申请时 Email 字段加提示："建议填写 Google 邮箱，方便后续一键登录"。

```
Google 登录匹配逻辑（按顺序）：
1. google_id 匹配 → 直接登录
2. Google 账号 email 匹配 users.email → 登录 + 自动写入 google_id
3. 都不匹配 → "这个 Google 账号还没有关联串门儿账号"
   → [用注册邮箱 magic link 登录] / [去申请加入]
```

已登录用户可在"我的页面 → 设置"中绑定 Google 账号：
- 点击"绑定 Google 账号"→ Google OAuth 授权 → 写入 `google_id`
- 绑定后 Google 一键登录和 magic link 都可用
- 一个 Google 账号只能绑定一个串门儿账号

**技术方案**：AWS Amplify Auth（Cognito），Google OAuth 2.0 + Email provider 双通道。

#### 身份层级

```
用户状态枚举 user_status:
  'applicant'     -- 已提交申请，等待审核
  'approved'      -- 审核通过，可登录
  'rejected'      -- 审核未通过
  'banned'        -- 被封禁

用户角色枚举 user_role（仅 approved 用户有角色）:
  'member'        -- 普通成员
  'admin'         -- 管理员（V1: Yuan / 白开水 / 大橙子）
```

**管理员指定**：现有管理员在管理后台 → 成员管理 → 指定角色 → 授权新管理员。V1 不限制管理员数量，但建议 ≤5 人。

**权限四层**（详见第十一章）：

| 层级 | 谁 | 能做什么 |
|------|-----|---------|
| 访客 | 未登录 / 未注册 | 浏览公开内容、提交申请 |
| 成员 | approved + member | 报名、投票、推荐、评论、发起活动、寄感谢卡 |
| Host | 成员中发起活动的人 | 自己活动的全部管理权限 |
| 管理员 | approved + admin | 管理后台全部操作 + 审核申请 + 指定管理员 |

### 5.3 评论

评论是所有内容类型的通用互动能力。V1 支持纯文字评论。

**权限规则**：

| 内容类型 | 谁能评论 | 说明 |
|---------|---------|------|
| 活动（已结束） | 仅该活动参与者（status='accepted' 的用户） | 活动回忆属于在场的人 |
| 电影 | 所有已登录成员 | 电影讨论是开放话题 |
| 提案 | 所有已登录成员 | 鼓励参与讨论 |
| 话题讨论 | 所有已登录成员 | 开放讨论 |

**评论交互**：
- 评论按时间正序展示（最早在上）
- 每条评论：头像 + 显示名 + 时间 + 内容
- 输入框在列表底部，placeholder 因类型不同："写点回忆..."（活动）/ "说说你的看法..."（电影/提案）
- 字数限制：500 字，超出时输入框边框变红 + 剩余字数提示
- 发布后即时出现在列表中，无需刷新

**评论来源标签**：

跨实体评论自动携带来源标签，帮助读者理解评论的上下文：
- 活动评论：如果该活动放映了某部电影，评论自动带"也评论了「{电影名}」→"标签（点击跳转电影详情页）
- 电影评论：如果评论者参加过放映该电影的活动，评论自动带"来自「{活动名}」→"标签（点击跳转活动详情页）
- 技术实现：发布评论时，后端检查关联关系自动写入 `comments.source_entity_type` 和 `source_entity_id`，前端渲染为可点击标签

**编辑与删除**：
- 评论者可编辑自己的评论（编辑后显示"已编辑"标记，不保留编辑历史）
- 评论者可删除自己的评论（软删除，显示"该评论已删除"）
- Host 可删除自己活动下的任何评论
- 管理员可删除任何评论（管理后台 → 评论管理）

**评论通知**：
- V1 不为评论单独发通知（避免邮件过多）
- 活动/电影/提案的评论数显示在对应卡片上（动态流、列表页）
- 新评论不触发动态流卡片浮顶（与话题讨论不同——话题讨论的核心就是讨论，活动/电影的核心不是）

**管理后台**：
- 评论管理列表：内容摘要、作者、关联实体、发布时间
- 批量删除、按实体筛选

### 5.5 Email 通知系统

Email 是 V1 的核心运营杠杆。通过 rule-based 逻辑在对的时机推一把——活动提醒、感谢卡引导、Host 培养、新人融入、沉默用户再激活。

**核心约束：每人每天最多收到 1 封邮件。** 事务性邮件（活动取消等不可控紧急事件）除外，但这类邮件极少触发。

#### 设计原则

- **每天一封**：所有非紧急内容合并进一封每日邮件，不轰炸
- 像朋友说话，不像平台通知
- 不制造紧迫感（不说"仅剩 X 个名额！"）
- 不催促（不说"你还有 X 张卡没用"）
- 不暴露经济机制（不提"退税""免费"等概念）
- 参考串门儿内容风格指南的"Mean + 懦弱 + 搞笑 + 真诚"

好的例子：
- "你投的《花样年华》被选中了，这周六在白开水家放映，要不要来？说实话我觉得你不来挺亏的"
- "上次你在 XX 家的电影夜，大家聊到快 12 点，好像很开心？要不要试试自己也办一次？不用完美，3-4 个人就够了"

不好的例子：
- "🔥 热门活动即将开始！名额有限，立即报名！"
- "尊敬的用户，我们注意到您最近未参与活动..."

#### 邮件分类

| 类型 | 说明 | 频率 | 占比 |
|------|------|------|------|
| **事务性邮件** | 不可延迟的运营关键信息，立即发送 | 极少，用户可能数周不收到一封 | ~5% |
| **每日邮件** | 所有非紧急内容合并成一封 | 每人每天最多 1 封 | ~95% |

两类邮件共享同一个 `email_log` 表做去重和追踪。

---

#### 事务性邮件（立即发送）

仅限以下 4 种——**必须立即送达**，等不到每日邮件。

| ID | 触发事件 | 收件人 | Subject | Body 要点 | 备注 |
|----|---------|--------|---------|----------|------|
| TXN-1 | 活动取消 | 所有已报名者 | "{event_name}已取消" | 取消原因/原活动时间/推荐 1 场同类型替代活动 | **任何邮件状态都发送**（含退订/stopped） |
| TXN-2 | 活动时间或地点变更 | 所有已报名者 | "{event_name}的{changed_field}改了" | 变更前→后对比/更新后完整信息 | **任何状态都发送** |
| TXN-3 | Waitlist 递补 | 递补人 | "{event_name}有位置了" | 活动信息/剩余确认时间（24h）/[确认参加] | 24h 未确认自动释放，不追发 |
| TXN-4 | 申请审核通过 | 新成员 | "欢迎来串门儿 👋" | 欢迎文案/串门原则摘要/登录链接（Google 登录或 magic link）/近期 2 场活动/[去看看] | 一次性 |
| TXN-5 | 申请未通过 | 申请者 | "关于你的串门儿申请" | 温和婉拒文案，不解释具体原因/"30 天后可重新申请" | 一次性 |
| TXN-6 | 小局抽签通知 | 被抽中者 | "🎲 轮到你做东了！" | "约谁、做什么、几个人，你说了算。"/[发起小局 →]/[这周不行 →] | 每周最多 1 次，跳过后重抽新人收到 |

**为什么这 4 个必须立即发**：
- TXN-1/2：影响用户是否出门、去哪里，延迟 = 白跑一趟
- TXN-3：有 24h 确认倒计时，等次日每日邮件可能已经过了一半时间
- TXN-4：申请通过是用户等待后的结果，立即通知 + 提供登录入口是基本预期

---

#### 每日邮件

##### 发送时间：两种模式

| 模式 | 触发条件 | 发送时间 | 说明 |
|------|---------|---------|------|
| **常规模式** | 默认 | 每天 09:00 AM EST | 大多数日子 |
| **活动日模式** | 用户今天参加了活动 | 活动结束后 2 小时 | 抓住散场后的情感高点 |

**活动日模式规则**：
- 如果用户今天参加了活动且活动已结束，2h 后触发活动日邮件
- 活动日邮件包含**所有常规每日邮件内容 + 活动后引导**（感谢卡模块自动排在最前面）
- 发送后标记 `today_daily_sent = true`，次日 9am 不再重复发
- 如果一天参加多场活动：以**最后一场**的结束时间 +2h 为准
- 如果活动结束时间 + 2h 晚于 23:00 EST：延迟到次日 8:00 AM 发送（不打扰睡眠）
- 如果活动结束时间 + 2h 早于 9:00 AM（如早上的户外活动）：延迟到 9:00 AM 合并发送

**为什么活动日模式重要**：感谢卡是核心机制，最佳引导窗口在散场后 1-2h。如果统一等到次日 9am，周六晚上 10 点散场 → 周日早上 9 点才收到引导 → 感觉已经凉了。活动日模式让引导出现在情感最浓的时候。

##### 数据定义

以下定义贯穿所有规则，开发时直接引用。

**用户标签（非互斥，一个用户可以同时拥有多个标签）**：

```
标签判定逻辑（每个标签独立计算）：

tag_new_user:        注册时间 ≤ 14 天
tag_post_event:      用户参加的某场活动在 scan_window 内结束
tag_active_non_host: 过去 90 天内参加 ≥3 场活动 AND Host 次数 = 0
tag_dormant_host:    曾 Host ≥1 次 AND 最近一次 Host > 21 天前
tag_lapsed:          过去 90 天内参加过 ≥2 场活动 AND 最近一次参加 > 21 天前
tag_first_event:     参加活动总次数 = 1 AND tag_post_event = true
```

**重要变更**：V1.8 使用互斥状态（命中即停），导致新人参加活动后无法触发 post_event 逻辑。V1.9 改为**非互斥标签**，每个标签独立判断。一个注册 3 天的新人参加了第一次活动，同时拥有 `tag_new_user` + `tag_post_event` + `tag_first_event`。

**"认识的人"定义**：

```sql
-- user_a 和 user_b 互相认识 ⟺ 同场参加过至少 1 场已结束的活动
SELECT DISTINCT es2.user_id
FROM event_signups es1
JOIN event_signups es2 ON es1.event_id = es2.event_id
JOIN events e ON e.id = es1.event_id
WHERE es1.user_id = {current_user}
  AND es2.user_id != {current_user}
  AND es1.status = 'accepted'
  AND es2.status = 'accepted'
  AND e.phase = 'ended'
```

**活动类型（从 tags 推断）**：

events 表不设独立 `event_type` 字段。活动类型从 `events.tags[]` 推断，后端提供 `get_event_type(tags)` 工具函数：

```
推断规则（按优先级，命中即返回）：
  tags 包含 '电影' → 'movie_night'
  tags 包含 '聚餐' or 'Potluck' or '火锅' → 'potluck'
  tags 包含 '徒步' or '户外' → 'outdoor'
  tags 包含 '聊天' or '桌游' or '小局' → 'social'
  兜底 → 'other'
```

**event_type_label 映射**（用于 Email 模板变量 `{event_type_label}`）：
```
movie_night → "电影夜"
potluck → "聚餐"
outdoor → "户外活动"
social → "聊天局"
other → "活动"
```

**注意**：所有 Email 规则中引用 `e.event_type` 的地方，实际实现为 `get_event_type(e.tags)`。

**用户偏好活动类型**：

```sql
-- 从用户参加过的活动 tags 推断偏好类型
SELECT get_event_type(e.tags) as derived_type FROM event_signups es
JOIN events e ON e.id = es.event_id
WHERE es.user_id = {current_user} AND e.phase = 'ended'
GROUP BY derived_type HAVING COUNT(*) >= 2
```

**扫描窗口**：

```
scan_window_start = MAX(
  user.last_daily_sent_at,
  NOW() - INTERVAL 24 HOUR
)
scan_window_end = NOW()
```

##### 内容优先级

每日邮件按 P0 → P4 顺序收集内容，组装成一封。

**P0 · 需要你回应的**

不限条数，永远展示。

| 规则 ID | 触发条件 | 模板 | CTA | 管理员可调 |
|--------|---------|------|-----|----------|
| P0-A | 有未响应的定向邀请（event_signups.status='invited'，扫描窗口内） | "{host} 邀请你参加「{event_name}」\n{date} {time} · {location}" | [接受邀请] [婉拒] | 模板文案 |
| P0-B | 已报名活动在 [NOW, NOW+24h] 内开始 | "你报名的「{event_name}」明天就是了\n{date} {time} · {location_display}\nHouse rules: {rules_summary}" | [查看详情] | 提前时间（默认24h） |

**P1 · 新活动推荐**

最多 3 场。

```
查询：
SELECT e.* FROM events e
WHERE e.phase = 'open'
  AND e.published_at IN scan_window
  AND e.start_time > NOW()
  AND e.id NOT IN (SELECT event_id FROM event_signups WHERE user_id = {current_user} AND status = 'accepted')
  AND e.id NOT IN (SELECT event_id FROM event_exclusions WHERE excluded_user_id = {current_user})

排序 score（管理员可在后台调整各权重）：
  +100  e.movie_id IN (用户投票过的电影 ID)
  +50   get_event_type(e.tags) IN (用户偏好活动类型)
  +25   e.participants 中有 ≥1 个用户认识的人
  +0    兜底 e.start_time ASC

取 TOP 3
```

**模板**（3 个变体随机选 1 个，管理员可编辑/新增/删除变体）：

```
变体 A: "{host} 发起了「{event_name}」\n{date} {time} · {location} · 还剩 {spots_left} 位"
变体 B: "{date}有场{event_type_label}，在{host}家，来不来？\n还剩 {spots_left} 位"
变体 C: "这周{host}开门了——{event_name}\n{date} {time} · 剩 {spots_left} 位"
```

**社交上下文注入**：
```
对每场推荐的活动：
  acquaintances = 活动 participants ∩ 用户认识的人
  IF acquaintances 不为空:
    取共同参加活动次数最多的 1 人
    附加行: "{name} 也报名了"
```

CTA: [报名参加]

**P2 · 感谢卡 & 称号**

最多 3 条（感谢卡优先）。

| 规则 ID | 触发条件 | 模板 | CTA | 管理员可调 |
|--------|---------|------|-----|----------|
| P2-A | 扫描窗口内收到感谢卡。如果 30 分钟内收到多张，合并为 1 条 | 单张："{sender} 给你寄了一张感谢卡\n\"{message 前 30 字}...\"\n—— 来自{event_name}" | [查看卡片] | 模板文案 |

| P2-B | 扫描窗口内获得新称号 | "你被大家封为「{title_name} {title_emoji}」" | [查看] | 模板文案 |

按以下顺序匹配，**命中第一条即停止**（管理员可调整顺序、启用/禁用每条规则）：

| 规则 ID | 前置标签 | 触发条件 | 冷却期 | 模板（按角色分） |
|--------|---------|---------|--------|----------------|
| P3-A | tag_post_event + tag_first_event | 第一次参加活动后 | 仅 1 次（lifetime） | **新人首次活动后**："第一次来串门，感觉怎么样？如果有谁让你觉得特别放松、特别被欢迎，可以跟 TA 说一句。\n[写一张感谢卡 →]\n这不是必须的，只是如果你想的话。" |

| P3-C | tag_new_user | 参加活动次数 = 0 | 3 天 | "欢迎来串门儿 👋 这周有 {upcoming_count} 场活动可以参加，去看看？\n[查看活动]" |
| P3-D | tag_active_non_host | — | 30 天 | "你已经来了 {attended_count} 次了，要不要试试自己办一次？不用完美，3-4 个人就够了\n[发起活动]" |
| P3-E | tag_dormant_host | — | 30 天 | "好久没见你开门了，最近忙？什么时候再来一场？\n[发起活动]" |
| P3-F | tag_lapsed | 本周有 ≥1 场公开活动 | 21 天 | 优先版（有社交上下文）："好久没见，{thanked_user}这周六在{host}家有个{event_type_label}，要不要来？\n[查看活动]"\n兜底版："好久没见你了，这周有个{event_type_label}你可能感兴趣\n[查看活动]" |

**P3 关键设计变更（对比 v1.8）**：

1. **P3-A/B 排序翻转**：活动后引导优先于新人欢迎。理由：如果一个新人刚参加完第一次活动，引导 TA 发感谢卡（P3-A）比重复说"欢迎来串门"（P3-C）更有价值。P3-A 专门为"新人第一次活动"设计了更温暖的模板。

2. **P3-B 按角色分三个模板**：Host 收到"辛苦了"版本，记录人（recorder）收到"照片靠你了 + 感谢卡"版本，普通参与者收到"谁让你印象最深"版本。v1.8 只有一个统一模板，Host 收到"昨天在你自己家怎么样"很奇怪。

3. **标签改为非互斥**：修复了 v1.8 中新人无法触发 post_event 引导的 bug。

4. **P3-F 加入社交上下文**：用感谢卡数据找用户之前感谢过的人，如果那个人报名了近期活动，推送里带上 TA 的名字。"XX 也会去"比"有个活动你可能感兴趣"有效得多。

```
P3-F 社交上下文查询：
  thanked_users = SELECT to_user_id FROM postcards WHERE from_user_id = {current_user}
  upcoming_with_thanked = SELECT e.* FROM events e
    JOIN event_signups es ON e.id = es.event_id
    WHERE es.user_id IN (thanked_users) AND es.status = 'accepted' AND e.phase = 'open' AND e.start_time > NOW()
    LIMIT 1
  
  IF upcoming_with_thanked:
    使用优先版模板，{thanked_user} = 该用户的 display_name
  ELSE:
    使用兜底版模板
```

**冷却期实现**：

```sql
SELECT 1 FROM email_log 
WHERE user_id = {current_user} AND rule_id = {rule_id}
  AND sent_at > NOW() - INTERVAL {cooldown_days} DAY
LIMIT 1
-- 有结果 → 跳过该规则
```

**P4 · 社群动态**

最多 1 条。**跳过条件**：P0 + P1 + P2 + P3 合计 ≥ 6 条时跳过。

| 规则 ID | 触发条件 | 冷却期 | 模板 | 管理员可调 |
|--------|---------|--------|------|----------|
| P4-A | 已完成活动总数跨过整十 | 该里程碑仅 1 次 | "串门儿第 {N*10} 场活动 🎉" | 里程碑步长（默认10） |
| P4-B | 扫描窗口内新增电影推荐 ≥ 2 部 | 7 天 | "本周新增 {N} 部电影推荐" | 最低推荐数（默认2） |
| P4-C | 每月 1 日 | 月度 | "上个月 {host_count} 位 Host 开放了自己的空间，感谢 {host_names}" | 模板文案 |

##### 邮件组装

**常规模式（9:00 AM）**：

```
Step 1: 获取所有 email_state != 'unsubscribed' AND != 'stopped' 的用户
Step 2: 排除 email_state = 'weekly' 且今天不是周一的用户
Step 3: 排除今天已通过活动日模式发送过的用户（today_daily_sent = true）
Step 4: 对每个用户，按 P0→P4 顺序执行触发规则，收集内容
Step 5: 如果内容为空，跳过
Step 6: 按截断规则组装
Step 7: 生成 Subject
Step 8: 发送，写入 email_log，更新 last_daily_sent_at
```

**活动日模式（活动结束 +2h）**：

```
触发: events.phase 变为 'ended'
延迟: event.end_time + 2h（如超过 23:00 延迟到次日 8:00）
收件人: 该活动的所有 participants（排除 unsubscribed/stopped）

Step 1: 收集该用户的 P0→P4 全部内容（和常规模式相同的规则）
Step 2: 如果 P3 命中了 P3-A 或 P3-B（活动后引导），将其提升到邮件最顶部（在 P0 之前）
Step 3: 组装、发送
Step 4: 标记 today_daily_sent = true，次日 9am 跳过该用户

如果一天参加多场活动：
  - 每场活动独立触发 P3-A/B 的检查
  - 但只发 1 封邮件：以最后一场结束时间 +2h 为准
  - 邮件中可包含多场活动的感谢卡引导（P3 上限从 1 条放宽到"每场活动 1 条"）
```

**截断规则**：

```
1. 如果有 P3-A 或 P3-B（活动后引导）：排在最前面，不截断
2. P0 全部内容：不截断
3. P1：取 TOP 3
4. P2：取 TOP 3（合并后的感谢卡算 1 条）
5. P3（非活动后引导）：total < 5 时取 1 条，否则跳过
6. P4：total < 6 时取 1 条，否则跳过
```

**邮件结构**：

```
[Subject Line]

─── 如果是活动日模式，且有活动后引导 ───
💬 [P3-A/B 感谢卡引导文案，按角色分]

─── 如果有 P0 ───
🔔 [P0 内容，每条一段，含 CTA]

─── 如果有 P1 ───
📅 [活动列表，每场一行]
   [社交上下文（如有）]
   [查看全部活动]

─── 如果有 P2 ───
✉ [感谢卡/称号，每条一段]

─── 如果有非活动后的 P3 ───
💡 [引导文案，含 CTA]

─── 如果有 P4 ───
📊 [社群动态，一行]

───────────────────────────
串门儿 · 退订 · 通知设置
```

##### Subject Line

取邮件中最高优先级内容，套模板（管理员可编辑所有模板）：

```
优先级匹配（命中即用）：

1. 有活动后引导（P3-A/B） → "今晚在{host}家" / "辛苦了 Host 🏠" / "第一次来串门，感觉怎么样？"（按角色）
2. P0 有定向邀请（P0-A）   → "{host} 邀请你参加{event_name}"
3. P0 有活动提醒（P0-B）   → "明天 {time} {event_name}"
4. P2 有感谢卡（P2-A）     → "{sender}给你寄了一张感谢卡" / "你收到了{N}张感谢卡"
5. P1 score ≥ 100          → "你投的《{movie_name}》{date}放映"
6. P1 有活动               → "这周有 {N} 场活动可以参加"
7. P2 有称号（P2-B）       → "你被大家封为「{title}」"
8. P3 有引导               → 引导模板前半句
9. 仅 P4                   → "串门儿本周动态"
```

活动日模式下 Subject 优先使用活动后引导（第 1 位），因为这是用户此刻最相关的内容。

**格式约束**：Subject ≤ 40 字符（中文约 20 字），不带 emoji（Subject 里），不带品牌名。

##### 事务性邮件 → 每日邮件去重

事务性邮件发出后写入 `email_log`。每日邮件生成时检查：

| 事务性邮件 | 每日邮件 | 去重 key |
|-----------|---------|---------|
| TXN-3（Waitlist 递补） | P0-C 中不再出现 | ref_id = waitlist_entry_id |
| TXN-4（审核通过欢迎） | P3-C 不再触发 | rule_id + user_id |

TXN-1/2（取消/变更）无对应每日邮件模块，无需去重。

##### 数据模型

**email_rules 表**（管理员可配置）：

```sql
CREATE TABLE email_rules (
  id            VARCHAR(10) PRIMARY KEY,  -- 'P0-A', 'P3-B', 'TXN-1' 等
  category      ENUM('transactional', 'daily'),
  priority_tier ENUM('P0', 'P1', 'P2', 'P3', 'P4'),
  display_order INT,                      -- 同 tier 内的排序（管理员可调）
  template_id   INT REFERENCES email_templates(id),
  trigger_sql   TEXT,                     -- 触发条件的 SQL 片段
  cooldown_days INT DEFAULT 0,            -- 冷却期天数，0=无冷却
  max_per_tier  INT,                      -- 该 tier 最多几条
  enabled       BOOLEAN DEFAULT true,     -- 管理员可禁用
  description   TEXT,                     -- 规则说明（给管理员看）
  created_at    TIMESTAMP DEFAULT NOW(),
  updated_at    TIMESTAMP DEFAULT NOW()
);
```

**email_templates 表**（管理员可编辑文案）：

```sql
CREATE TABLE email_templates (
  id            SERIAL PRIMARY KEY,
  rule_id       VARCHAR(10) REFERENCES email_rules(id),
  variant_key   VARCHAR(20),              -- 角色/变体标识：'host', 'newcomer', 'regular', 'A', 'B', 'C'
  subject_tpl   VARCHAR(200),
  body_tpl      TEXT,                     -- 支持 {variable} 占位符
  is_active     BOOLEAN DEFAULT true,
  created_at    TIMESTAMP DEFAULT NOW(),
  updated_at    TIMESTAMP DEFAULT NOW()
);
```

**email_log 表**（追踪和去重）：

```sql
CREATE TABLE email_log (
  id            SERIAL PRIMARY KEY,
  user_id       INT NOT NULL,
  email_type    ENUM('transactional', 'daily_regular', 'daily_event') NOT NULL,
  rule_id       VARCHAR(10),
  ref_id        INT,                      -- 关联对象 ID（event_id / card_id 等）
  subject       VARCHAR(200),
  sent_at       TIMESTAMP NOT NULL,
  opened_at     TIMESTAMP NULL,
  clicked_at    TIMESTAMP NULL,
  click_count   INT DEFAULT 0,
  created_at    TIMESTAMP DEFAULT NOW()
);

-- 去重索引
CREATE INDEX idx_email_dedup ON email_log(user_id, rule_id, ref_id, sent_at);
```

**email_rule_params 表**（管理员可调的参数）：

```sql
CREATE TABLE email_rule_params (
  id            SERIAL PRIMARY KEY,

  param_key     VARCHAR(50),              -- 'score_weight_movie', 'cooldown_days', 'merge_window_min' 等
  param_value   VARCHAR(100),
  description   TEXT,
  created_at    TIMESTAMP DEFAULT NOW()
);
```

```
┌─────────────────────────────────────────────────────────┐
│                    定时任务                               │
│                                                         │

│  9. 写入 email_log                                      │
└─────────────────────────────────────────────────────────┘
```

#### 用户邮件状态机

```
                 ┌──────────────────────────────────────┐
                 │                                      │
                 ▼                                      │
  ┌─────────┐  注册   ┌────────┐  连续3封   ┌────────┐  │  连续6封   ┌─────────┐
  │ 未注册   │ ─────→ │ active │ ─────────→ │ weekly │ ─┤ ────────→ │ stopped │
  └─────────┘        └────────┘  未打开     └────────┘  │  未打开    └─────────┘
                        ▲   ▲                   │       │              │
                        │   │     打开/点击/登录  │       │              │
                        │   └───────────────────┘       │              │
                        │         打开/点击/登录          │              │
                        └───────────────────────────────┘              │
                        │                                              │
                        │              用户手动重新订阅                   │
                        └──────────────────────────────────────────────┘

  另有独立状态：
  ┌──────────────┐
  │ unsubscribed │  ← 用户点击退订
  └──────────────┘  → 用户在偏好页重新开启 → active
```

**状态定义**（管理员可调整阈值）：

| 状态 | 每日邮件 | 事务性邮件 | 触发条件 |
|------|---------|-----------|---------|
| `active` | 每天发 | 全部发 | 默认 |
| `weekly` | 仅周一发 | 全部发 | 连续 3 封未打开（可调） |
| `stopped` | 不发 | 仅 TXN-1/2（取消/变更）| 连续 6 封未打开（可调） |
| `unsubscribed` | 不发 | 仅 TXN-1/2 | 用户退订 |

**恢复逻辑**：打开邮件 / 点击链接 / 登录网页端 / 手动开启 → 重置为 `active`。

#### 免打扰时段

```
用户设置 quiet_hours_start 和 quiet_hours_end（如 22:00-08:00）。

每日邮件：
  常规模式：如果 09:00 在免打扰内 → delay 到 quiet_hours_end

  TXN-1/2（取消/变更）：无视免打扰

```

- V1 的规则表和模板库直接作为 Agent playbook
- Agent 输出运营日报（社群健康度、推送统计、运营建议）
- 初期 Agent 输出经人工审核，稳定后自动发送 + 抽检

### 5.6 人与人的连接层

这是设计原则「关系优先于活动」的核心实现。三个机制，从被动到主动：

#### 共同品味与共同记忆（被动展示）

当用户访问另一个成员的个人页面时，系统自动展示两人之间的关联：

```
你和 XX 的共同经历：

🎬 共同品味
  - 你们都投了：《花样年华》《惊魂记》《永恒和一日》
  - 你们都喜欢：文艺片、日本电影

📅 一起参加的活动
  - 12/14 白开水家电影夜
  - 1/5 新年 Potluck
  - 1/20 Central Park 徒步
```

实现方式：纯数据交叉计算，不需要额外用户输入。**这不是推荐算法，是事实展示。**它把模糊的"我好像在哪见过你"变成具体的共同记忆。

#### 感谢卡（主动连接）

活动结束后，Email 系统引导参与者发送感谢卡。感谢卡是串门的核心社交货币——比点赞重，比私信轻。同时也是 V2 会员经济模型的基础数据源（见九、未来方向）。V1 支持免费获得 + $5/张购买两种获取方式。

**额度获取规则**：

| 来源 | 额度 | 说明 |
|------|------|------|
| 新人注册 | 4 张 | 审核通过后自动赠送，体验完整机制 |
| 参加活动 | +2 张/次 | 到场即获得 |
| Host 活动 | +4 张/次 | 核心贡献额外奖励 |
| 花钱购买 | $5/张 | 不限量，随时可买 |

**感谢卡购买支付流程**：
```
用户点击"购买感谢卡"→ 选择数量（1/3/5/10 张）
  → 跳转 Stripe Checkout（或内嵌 Stripe Elements）
  → 支付成功 → Stripe Webhook 回调
  → 后端写入 postcard_credits（source='purchase'）+ postcard_purchases
  → 前端更新余额显示
```
支付失败或取消：返回原页面，余额不变。退款：Stripe Dashboard 手动处理，后端扣回对应额度。

**支付后草稿恢复**：用户从感谢卡页面跳转 Stripe 前，前端将寄卡草稿（收件人 ID、消息内容、照片、可见性选择）存入 `sessionStorage`。Stripe 回调返回后自动恢复草稿内容，用户可直接寄出。`sessionStorage` 在页面关闭后自动清除，不做长期持久化。

**额度规则**：
- 累计不清零，无过期时间
- 显示当前余额（如"可寄 X 张"），但不显示过期时间（因为不过期）
- 无月度上限

**发送范围限制**：
- **免费获得的卡**（参加活动/Host/新人赠送）：只能发给同场活动的参与者
- **花钱购买的卡**（$5/张）：可以发给社群里的任何人

这条规则堵死了"不见面互刷"的作弊路径。想给不在同场的朋友发卡（比如生日祝福），需要花 $5——这是慷慨行为，不是作弊。

**隐私模式**：

| 模式 | 说明 | 展示位置 |
|------|------|---------|
| 🌐 公开 | 所有人可见 | 动态流 + 收件人个人页面 + 感谢卡页 |
| 🔒 私密 | 仅收件人可见 | 收件人个人页面（带 🔒 标记） |

发送时默认公开，发送者可选择改为私密。

**给 Host 的感谢卡**：
- 提供预设标签一键选择："好吃！""氛围很棒""下次还来""Host 太用心了"
- 可选：展开写一句话
- 可选：附上活动场景照片（系统提供渐变氛围图兜底）
- Host 收到通知 + 感谢卡展示在个人页面"收到的感谢卡"区域

**给同场参与者的感谢卡**（更重要）：
- Email 推送引导："今天谁让你印象最深？给 TA 说句话"
- 选一个人 → 写一句话或选预设标签："聊得很开心""很有趣的人""下次还想见到你"
- 收件人收到通知："XX 在「XX 活动」后给你寄了一张感谢卡"

**感谢卡页**（✉ Tab）：
- **单页布局，无 Tab 切换**
- 寄卡流程：三步进度条（选人 → 写话+附照片+选可见性 → 预览寄出）
- **附照片说明**：从手机相册选择 1 张照片（非拍照），支持 jpg/png/heic，≤5MB，上传后自动压缩。照片可选，不附照片时系统使用活动场景渐变图兜底
- 选人列表默认展示最近同场参与者，购买的卡可选任意成员
- 顶部余额显示："✉ 可寄 X 张"（无过期提示）
- **额度不足处理**：额度为 0 时，"寄出"按钮置灰，下方显示："额度不足 · [购买感谢卡 $5/张 →]"。点击跳转 Stripe 购买流程，支付成功后自动返回寄卡页面继续操作
- 额度为 0 的首屏引导："参加活动获得更多额度"或"购买感谢卡 · $5/张"
- 收到的感谢卡：标题"收到的感谢卡 · X 张"，支持展开/叠放视图切换
- **空状态**：新用户看到引导文案"参加一次活动后，就可以给同行的人寄一张感谢卡"+ 跳转活动页按钮

**感谢卡 Email 引导**（对应 Email 通知系统中的具体规则）：

| 触发 | 对应规则 | 内容策略 | 禁区 |
|------|---------|---------|------|
| 活动后引导寄卡 | P3-A/B（活动日邮件，散场后 2h） | 按角色变化模板（新人首次/Host/参与者），底部永远有"不想写也没关系" | 不催促、不提额度 |
| 收到卡通知 | P2-A（每日邮件，30 分钟合并窗口） | 通知内容 + 发送者 + 来源活动。**不引导回发** | 引导回发 = 制造互刷 |
| 称号获得 | P2-B（每日邮件） | 惊喜通知，不解释触发规则 | 不暴露阈值 |
| 沉默用户再激活 | P3-F（冷却 21 天，带社交上下文） | 用感谢卡数据找 TA 之前感谢过的人，推荐该人参与的活动。推送里不提卡 | 冷却 21 天 |
| 付费提醒 | V2 新增规则 | 列活动 + 休眠按钮，不说"参加了就免费" | 不暴露退税逻辑 |

**推送频率控制**：活动后引导每场最多 1 次（P3-A/B 冷却），收到卡通知随每日邮件合并发送（P2-A），称号通知随每日邮件发送，再激活 P3-F 冷却 21 天。

为什么重要：
- 它把"我们一起参加了一个活动"变成"你专门对我说了一句话"
- 私密模式让人更愿意表达真实感受
- 公开模式让社群看见感恩的文化，鼓励更多人表达
- 它创造了活动之外的人际信号，是关系加温的起点
- 数据层面：谁给谁发过感谢卡 = 一种隐性的关系强度指标
- 称号系统的输入源：感谢卡的预设标签就是称号的"选票"（见 5.6）
- V2 经济层面："被多少独立发送者发过卡"是贡献者判定的数据源之一

#### 社交上下文（后端数据层，非前端 UI）

社交上下文是后端通过数据交叉计算生成的关系信息，**不是前端页面上的独立 UI 模块**。它为其他功能提供数据支撑：

**计算内容**：
- 两人共同参加过几次活动
- 两人是否都投过同一部电影
- 某用户是否收到过对方的感谢卡
- 某成员是否是该用户"从未见过的新面孔"
- 某成员是否在上次活动中待到了最后

**数据消费方**：
- **Email 推送**：规则引擎在组装推送内容时参考社交上下文（如"推送有想再见标记的人也参加的活动"）
- **Host 定向邀请建议**：Host 创建活动时，系统基于数据提供邀请建议（如"以下成员参加过你上次的活动且待到了最后"）
- **成员详情页"共同经历"**：展示两人之间的事实关联（见上文"共同品味与共同记忆"）

**不在前端直接展示的场景**：
- ~~活动详情页的社交上下文卡片~~（已移除，属于后端数据能力，不是前端 UI 需求）

**实现原则**：
- 纯数据交叉查询，不是推荐算法
- 展示事实（"你们一起参加过 3 次活动"），不做判断（"你们关系很好"）
- 数据计算在后端完成，前端只负责渲染后端返回的结构化结果

#### 集体里程碑（社群归属感）

在动态流中穿插社群整体的成就，让人感受到"我是一个正在成长的集体的一部分"：

- "串门第 50 场活动 🎉 从去年 10 月到现在，我们一起看了 XX 部电影、吃了 XX 顿饭、走了 XX 次路"
- "电影池突破 100 部！最受欢迎的 3 部是..."
- "本月 8 位 Host 开放了自己的客厅，感谢 XX、YY、ZZ..."
- "XX 和 YY 从电影夜认识，现在成了室友"（需要当事人同意）

由运营手动发布。

#### 关系数据的使用原则

- **所有关系信号都是私密的**：谁给谁发了私密感谢卡，只有当事人知道
- **系统展示的是事实，不是判断**：显示"你们一起参加过 3 次活动"，不显示"你们关系很好"
- **不做社交图谱排名**：不展示"最受欢迎的人"、"连接最多的人"，避免社交压力
- **服务于线下连接**：所有线上的关系信号，目标都是促成下一次线下见面

### 5.7 Host 激励体系

好的 Host 是社群的发动机。用一切手段让 Host 感到被看见、被感谢、被回馈。

**核心理念**："贡献的人不花钱，搭便车的人付钱。"

#### Host 可见性

Host 的贡献在平台上被充分展示：
- **成员墙 & 个人页面**：显示 Host 次数、Host 过的活动列表
- **活动记录**：每场活动的 Host 被突出展示
- **动态流**：Host 发起活动时，动态卡片突出 Host 身份

#### 荣誉体系

| 机制 | 说明 | V1 是否实现 |
|------|------|------------|
| **Host 徽章** | 个人头像旁显示 Host 标识（如小房子图标） | ✅ V1 |
| **Host 次数里程碑** | 首次 Host 🏠、5 次 ⭐、10 次 🔥、20 次 👑 | ✅ V1 |
| **月度 Host 致敬** | 每月动态流置顶"本月感谢这些 Host"（规则自动生成，运营确认） | ✅ V1 |
| **年度 Host 致敬** | 年度感谢（侧重"感谢这些人"而非排名竞争） | ⚠️ 谨慎 |

#### 称号与身份系统

称号不是自己给的，是别人给的。主打亲切好玩，不搞稀缺。

**社群称号（来自感谢卡积累）**

感谢卡（见 5.5）的预设标签就是称号的"选票"。累积收到同类标签达到一定次数，自动获得对应称号，显示在个人页面和成员墙头像旁。

| 标签方向 | 对应称号举例 | 触发条件（参考） |
|---------|------------|----------------|
| "好吃！""做饭太厉害了" | 🍳 厨神 | 累计收到 3 次 |
| "选片好""品味太好了" | 🎬 选片人 | 推荐的电影被放映 3 次 |
| "氛围很棒""很会聊天" | 🔥 氛围担当 | 累计收到 3 次 |
| "第一次来就很放松" | 🧊 破冰王 | 有新人的活动中累计收到 3 次 |
| "每次都帮忙收拾" | 🧹 收拾之神 | 累计收到 3 次 |
| "拍照太好了" | 📸 首席摄影 | 累计收到 3 次 |

- 一个人可以同时拥有多个称号
- 称号永久保留，越攒越多
- **阈值对用户完全不可见**：用户不知道触发规则，获得称号时是惊喜通知（"大家封你为厨神了 🍳"），不解释具体触发条件。防止用户算账、要卡
- 阈值由运营在管理后台配置和调整（见 11.1 称号管理）
- 新称号获得时在动态流生成一条卡片："XX 被大家封为「厨神」🍳"

**运营身份（运营指定）**

社群志愿者的职责标识，由运营在后台指定，显示在个人页面和成员墙。

| 身份 | 说明 |
|------|------|
| 🎨 设计总管 | 负责海报、视觉 |
| 📱 社媒小助手 | 负责小红书等对外内容 |
| 📷 活动摄影师 | 负责活动记录拍摄 |
| ✍️ 内容主笔 | 负责 Newsletter / 活动回顾 |
| 🔧 技术支持 | 负责平台相关 |

运营身份可随时调整，不是永久的。有运营身份的人同时也可以拥有社群称号。

#### 实质回馈

| 机制 | 说明 | V1 是否实现 |
|------|------|------------|
| **会员费减免** | Host 达到一定次数，会员费抵扣或免除 | 🔜 会员制上线后 |
| **贡献者免费** | 活跃 Host 自动进入"贡献者"身份，不需要付费 | 🔜 会员制上线后 |
| **优先报名** | Host 报名其他人的活动时享有优先权 | ✅ V1 |
| **专属活动** | Host-only 的交流活动（互相学习、吐槽、支持） | 线下运营，平台记录 |

#### Host 培养路径

系统不仅激励现有 Host，还要系统性地把参与者培养成 Host。Email 系统（P3-D、P3-E）负责识别时机和推送，以下是完整路径：

```
参与者 → 活跃参与者 → Co-Host → 独立 Host → 多类型 Host → 资深 Host

  参加 1 次：了解串门是什么
  参加 3 次：P3-D 规则建议 co-host + 自动推荐搭档（见下方 Co-Host 配对）
  Co-Host 1 次：P3-D 规则建议独立 Host
  独立 Host 1 次：获得 Host 徽章 🏠
  首次 Host 某小众类型：V2 Agent 可引导"要不要变成定期活动？"
  Host 5 次：里程碑 ⭐ + 会员费减免
  Host 10+ 次：资深 Host 🔥 + 贡献者身份（免费）
```

**Co-Host 自动配对机制**：

当 P3-D 规则识别到潜在 Host 时，不只推送"要不要试试"，而是主动匹配：
- 从活跃 Host 中选一个有经验的人
- 同时向双方推送："XX 想试试 co-host，YY 你愿意带 TA 一起办一次吗？"
- 双方同意后进入 co-host 模式

#### Host 保护

Host 愿意打开家门，平台必须保护这种信任：

- **静默排除**：创建活动时可设置"不可见名单"（见 4.4）
- **定向邀请优先**：两阶段机制让 Host 先锁定想邀请的人（见 4.4）
- **报名审核权**：公开阶段 Host 仍可审核报名
- **多次拒绝上报**：一个人被多个 Host 拒绝时，系统标记并通知运营介入
- **Host 求助不丢人**：活动详情页的"Host 求助"区域让协作成为常态
- **活动后匿名反馈**（V2）：让 Host 知道哪里可以改进，但不公开

| 部署 | AWS Amplify Hosting | 前端 + API（Lambda Functions） |
| 域名 | chuanmener.club | 已获取 |
| 定时任务 | AWS EventBridge + Lambda | Email 每日发送、小局抽签、Waitlist 超时 |
| 语言 | TypeScript（全栈） | 前后端统一语言 |

**MongoDB 数据建模说明**：下文"核心数据表"以关系型术语描述字段和关联。MongoDB 实际实现时：
- 一对一嵌套：`user_preferences` 嵌入 `users` 文档
- 一对多引用：`event_signups`、`postcards` 等高频查询用独立 collection + 索引
- 多对多引用：`movie_votes`、`likes` 等用独立 collection
- 聚合查询：动态流聚合用 MongoDB Aggregation Pipeline 替代 SQL UNION

### 总体数据策略

V1 使用独立数据库（MongoDB Atlas），不依赖 Notion API 作为运行时数据源。Notion 退役为运营后台备份和内容编辑工具。

**为什么不继续用 Notion 做数据源**：
- Notion API 读写速度慢，不适合实时交互
- 数据模型受限于 Notion 的表结构，难以表达复杂文档关联
- 感谢卡、话题讨论、协作者等新功能无法用 Notion 建模
- MongoDB 支持灵活文档结构、全文搜索、聚合管道

### 双向链接架构

系统中存在以下双向数据关联，前端通过跳转链接实现页面间的双向导航：

| 关联 | 关联方式 | 前端体现 |
|------|---------|---------|
| 电影 ↔ 活动 | movie_screenings 表关联 movie_id 和 event_id | 活动详情→电影卡片"查看电影详情→"；电影详情→放映记录"查看活动详情→" |
| 评论 ↔ 来源实体 | comments 表的 source_entity_type/id | 活动评论带"也评论了「电影名」→"标签；电影评论带"来自「活动名」→"标签 |
| 提案 ↔ 活动 | proposals 表 status=已排期 时关联 event_id | 提案卡片显示已创建的活动链接 |
| 成员 ↔ 活动/电影 | event_signups / movie_votes / event_roles 表 | 成员详情页展示参与的活动、推荐的电影、贡献角色 |

**实现原则**：
- 所有双向链接通过数据库关联实现，前端渲染为可点击的跳转链接
- V1 不实现评论间的跳转交互，仅展示来源标签
- 跳转链接样式统一为暖色文字 + → 箭头

### 社交上下文计算（后端数据层）

社交上下文不存储为独立表，而是运行时从关系数据交叉查询生成。

**计算内容**：

| 计算项 | 查询逻辑 | 数据消费方 |
|--------|---------|-----------|
| 两人共同参加的活动数 | event_signups 交叉 JOIN | 成员详情页"共同经历"、Email 推送 |
| 两人共同投票的电影 | movie_votes 交叉 JOIN | 成员详情页"共同品味" |
| 是否互发过感谢卡 | postcards 表查询 | Email 推送逻辑 |
| 是否为"新面孔" | 两人无任何 event_signups 交叉 | Email 推送逻辑 |
| Host 邀请建议 | 参加过上次活动 + 待到最后的成员 | Host 定向邀请流程 |

**不在前端直接展示的场景**：活动详情页不显示社交上下文卡片。社交上下文是后端数据能力，为 Email 推送、Host 建议、成员详情页提供数据支撑，不作为独立前端 UI 模块呈现。

### 动态流生成

动态流不存储为独立表，而是运行时从各数据表聚合生成。聚合逻辑：

```
SELECT * FROM (
  SELECT 'movie' as type, ... FROM movies WHERE created_at > ?
  UNION ALL
  SELECT 'event' as type, ... FROM events WHERE created_at > ?
  UNION ALL
  SELECT 'postcard' as type, ... FROM postcards WHERE visibility = 'public'
  UNION ALL
  SELECT 'discussion' as type, ... FROM discussions
  UNION ALL
  ...
) ORDER BY created_at DESC
```

规则生成的置顶内容（里程碑、Host 致敬等）由 Email 规则引擎（P4 层）触发，同时写入动态流。动态流聚合 SQL 中，里程碑和运营公告通过 `announcements` 表或 `feed_pinned` 标记实现置顶。

### Notion / Tally 数据迁移

V1 启动前一次性从 Notion 导入历史数据（成员、电影、活动记录），从 Tally 导入历史申请数据（已通过的申请者直接导入为 approved 成员，字段映射见 5.1 申请表字段表），之后 Notion 和 Tally 不再作为数据源。串门原则、Host 指南等内容在管理后台富文本编辑器（支持 Markdown 快捷输入）中直接维护，不再依赖 Notion 同步。

### 认证

- 申请制注册（见 5.1）：提交申请 → 管理员审核 → 通过后可登录
- **Google OAuth 2.0**（主要）：先查 `google_id`，再查 email 匹配。首次 Google 登录自动写入 `google_id`
- **Email magic link**（备选）：有效期 15 分钟，session 保持 30 天
- 已登录用户可在设置中绑定/解绑 Google 账号
- 技术方案：AWS Amplify Auth（Cognito），Google federated sign-in + custom email challenge

## 八、成功指标

### 核心指标（验证 V1 假设）

- **"我的页面"访问频率**：成员是否会回来看自己的参与记录？
- **投票/推荐/提案的活跃度**：网页端上线后，参与贡献的人数是否增加？
- **日/周活跃用户数**：是否真的"每天想来看看"？

### 人与人连接指标

- **感谢卡发送数**：活动后有多少人给 Host 或同场参与者发了感谢卡？
- **感谢卡公开比例**：有多少人选择公开感谢卡？（衡量社群信任度）
- **双向感谢卡比例**（观察性指标，不优化）：互相发过感谢卡的成员对数占总发卡对数的比例
- **话题讨论参与度**：每个话题的平均回复数
- **共同活动深度**：平均每对成员一起参加过几次活动？是否在增长？
- **跨活动类型连接**：成员是否在不同类型的活动中遇到同一个人？
- **成员页面互访频率**：看别人个人页面的频次

### Host 供给指标

- **月度活跃 Host 数**：每月至少发起 1 次活动的 Host 数量
- **新 Host 转化率**：收到引导后发起第一次活动的比例
- **Host 复做率**：Host 过 1 次的人中，3 个月内再次 Host 的比例
- **活动供给满足率**：Waitlist 比例越高说明供给越不足

### Email 效果指标

- **推送打开率**：Email 推送的打开率（按规则 ID 分组）
- **推送行动转化率**：打开后完成引导动作的比例（按规则 ID 分组）
- **降频率**：转为 weekly 或 stopped 的用户比例
- **退订率**：按周统计

### 辅助指标

- 电影投票数对比 Notion 时期是否增长
- 活动报名从网页端完成的比例
- 新提案的提交频率
- 点赞和评论数量

- **社群数据可视化**：整体的活动数量、参与人次、电影放映数等
- **AI Agent 升级**：从 rule-based Email 升级为 AI Agent。模板文案 → 个性化生成、固定规则 → 综合判断、运营日报自动生成、站内通知中心、A/B 测试推送文案
- **Host 匿名反馈**：活动后参与者匿名评价体验
- **会员制与经济模型**：

  **核心规则**：
  - 会员费 $30/月，所有人默认需要交
  - 本月被任何人发过感谢卡 → 下月免费（一张即可）
  - 手动休眠 → $0/月，但不能报名活动，随时可恢复
  - 连续活跃 3 个月 → 自动获得 1 个月宽限期（不来也不收费）

  **贡献者判定**：不设独立的"贡献者"身份判定。退税规则极其简单——被发过卡就免费。结果上，所有活跃参与者自动免费，只有"加入了但从不来"的人付费。

  **收入来源**：
  - 感谢卡购买（$5/张，价格不敏感者的自愿消费）
  - 会员费（不活跃用户的归属感税）
  - 服务收入（移民咨询、职业辅导等）
  - 企业合作（活动赞助、招聘等）

  **经济循环逻辑**：
  ```
  参与活动 → 获得感谢卡额度 → 发给值得感谢的人
                                    ↓
                            被发卡的人下月免费
                                    ↓
                  活跃者永远免费 ← → 不来的人交 $30 归属感税
                                    ↓
            想给不在同场的人发卡 → 花 $5 买 → 平台收入
  ```

  **V1 已为 V2 埋的数据**：感谢卡的"被多少独立发送者发过卡"、额度获取和消耗记录、称号触发数据。V2 上线时可直接使用。

## 十、视觉设计风格

### 风格定位：深色 Lofi × 现代日式 × 社区工具

**一句话**：深色底做空间感，层级做立体感，lofi grain 做质感，单暖色做呼吸。这是一个社区工具，不是杂志也不是 SaaS 后台。

### 调性关键词

**是**：深色的、有层次的、克制的、有噪点的、工具感的、现代的
**不是**：古朴的、手工感的、暖白的、多彩的、卡通的、过度设计的

### 配色

全站只用**灰阶层级 + 一个暖色强调**，不引入第二个色相。

**背景层级**（通过色阶做立体感，不靠阴影）：
- 底层背景：`#0C0C0E`
- Surface 1（卡片）：`#141416`
- Surface 2（浮层/elevated）：`#1C1C1F`
- Surface 3（hover / active）：`#262629`
- 分割线：`#2A2A2F`

**唯一强调色**：暖金 `#D4A574`——用于日期标记、主要按钮、感谢卡背景光晕、关键操作。低饱和变体 `#D4A57433` 用于底色晕染。除此之外不引入其他色相。

**文字**：
- 主文字：`#E8E6E2`（暖白，不是纯白）
- 副文字：`#9A9894`
- 弱文字/占位：`#5A5854`

### 质感与纹理

**全局 Grain 覆盖**：Canvas 生成的噪点纹理覆盖全屏，混合模式 `screen`（深色底上噪点是亮的，像胶片颗粒感）。opacity 约 0.35-0.5。这是 lofi 感的核心来源。

**磨砂玻璃导航**：顶部 sticky 导航用 `backdrop-filter: blur(12px)` + 半透明背景 `${s1}cc`，增加深度层次。

**照片处理**：活动照片可叠加轻微降饱和 + grain，和整体质感融合。头像保持原色不处理。

**无阴影设计**：不用 box-shadow 做立体感。层次完全通过背景色阶区分——底层暗、卡片稍亮、浮层更亮。

### 排版

**布局**：
- 现代日式留白，内容不挤。卡片间距 12px
- 统一圆角 10-12px
- 移动端单列，桌面端最多双列
- Sticky 磨砂导航栏

**字体**：
- 中文：系统默认（PingFang SC / Hiragino Sans / Noto Sans CJK）
- 英文：系统 San Francisco / -apple-system
- 全站无衬线体，不用手写体或装饰字体（**例外**：感谢卡组件使用衬线字体模拟纸质质感）
- 正文 14px，行高 1.7，深色底上稍小字号更精致

**标题层级**：
- 大标题 17-22px semibold，副标题 15px regular
- 用字重区分层级，不堆字号差异

### 内容呈现

**人是主角**：
- 头像 28-34px，灰色底（`#262629`），浅色文字
- 头像组叠放展示参与者（-6px overlap），带 `+N` 计数
- 活动卡片优先展示 Host 头像 + 称号标签

**照片区域**：
- 使用渐变色块代替无照片占位（`s2` → `s3` 渐变 + accent 微光晕）
- 有照片时叠加底部渐变遮罩，标题压在图上
- 日期标签用磨砂 pill 浮在照片右上角

**电影海报**：
- 海报区域用深色渐变底，保持整体暗调统一
- 投票按钮独立，不叠在海报上

**标签（Tag）**：
- 背景 `#262629`，文字 `#9A9894`
- 小巧不抢眼，圆角 5px

**感谢卡**：
- 唯一使用暖纸色底（`#F5F0E8`）和衬线字体（Georgia / Noto Serif SC）的组件
- 模拟真实感谢卡质感：邮票组件、手写风排版、带齿孔边缘
- 公开感谢卡可带照片（场景渐变图），私密感谢卡带 🔒 标记
- 和深色主界面形成鲜明物理质感对比，强调"这是特别的"

### 交互感受

**轻快不沉重**：
- 页面切换淡入淡出（200-300ms）
- 点击反馈用 opacity 变化（0.7），不用 scale
- 加载状态用骨架屏，颜色接近 `s1`，不刺眼

**微动效克制使用**：
- 感谢卡发送成功 → 轻微 confetti 或心形动画（唯一的"花哨"动效）
- 投票成功 → 数字 +1 轻微弹跳
- 其他操作只用状态变化

**声音**：无。

### 与现有物料的衔接

- 海报噪点纹理 → 网页 grain overlay（从纸质噪点变为胶片颗粒，深色适配）
- 海报排版 → 活动详情页可参考类似布局
- 照片预设 → 网页端照片沿用同一套调色，叠加 grain 融入深色底

### 设计系统优先级

V1 先定义以下核心组件的视觉规范：
1. 动态流卡片（8 种类型：系统横幅、里程碑、活动、感谢卡、活动回顾、电影推荐、提案、话题讨论）
2. 成员头像组件（头像 + 名字 + 称号标签 + 头像组叠放）
3. 按钮（主要 filled / 次要 outline / 文字链接 / 投票计数）
4. 电影海报卡片（渐变底 + 投票按钮）
5. 活动报名卡片（场景渐变图 + 两阶段邀请标识）
6. 感谢卡组件（暖纸色底 + 衬线字体 + 邮票 + 公开/私密标识）
7. 本月贡献卡片（事实列举，不计分）
8. Segment tab 切换器（圆角底 + 高亮当前项）
9. 空状态页面

其他组件在开发过程中迭代补充。

## 十一、权限体系

串门儿的权限分三层：管理后台（运营团队）、Host 活动权限、用户侧权限。V1 阶段运营团队只有 3 人（Yuan、白开水、大橙子），管理后台暂不分角色，所有管理员共享全部权限。团队扩大后再拆分管理员/运营两级。

### 11.1 管理后台

管理后台是运营团队的操作中心，网页端独立入口（汉堡抽屉 → 管理后台），仅管理员身份可访问。

**页面结构**：管理后台首页为 Tab 切换布局，包含以下模块：

#### 11.1.1 成员管理

**申请审核**：
- 待审核队列：按提交时间排序，显示申请人姓名、城市、自我介绍摘要、提交时间

- 操作：通过（发送 TXN-4 欢迎邮件 + magic link）/ 拒绝（发送婉拒邮件）

**成员列表**：
- 全量成员表格，列：头像、显示名、状态（approved/banned）、角色（admin/member）、运营身份标签、城市、加入时间、最近活跃时间、参与活动数、Host 次数、本月贡献摘要
- 排序：按活跃时间/加入时间/Host 次数
- 筛选：全部 / 活跃 / 沉默（3 周+未参与）/ 有问题标记

| 操作 | 说明 | 触发方式 |
|------|------|---------|
| 查看成员详情 | 跳转成员档案，含完整参与统计 | 点击行 |

#### 11.1.2 活动管理

#### 11.1.3 内容管理

**电影管理**：

#### 11.1.4 感谢卡管理

| 操作 | 说明 |

### 11.2 Host 活动权限

Host 对自己发起的活动拥有以下权限。所有 Host 权限仅限于自己发起的活动，不影响其他活动。

#### 报名阶段

| 权限 | 说明 |
|------|------|
| 定向邀请 | 第一阶段选择想邀请的人，被邀请人收到私人通知 |
| 不可见名单 | 设置哪些成员看不到这场活动（比"拒绝"更体面，对方不知道活动存在） |
| 审核报名 | 公开阶段接受/拒绝报名申请 |
| 移除已报名者 | 从已报名列表中移除某人（触发通知） |
| 管理 Waitlist | 查看排队者，手动递补 |

#### 活动内容

| 权限 | 说明 |
|------|------|
| 编辑活动 | 修改名称、时间、地点、人数上限、描述、house rules |
| 选片 | 电影夜活动从候选池中选定放映影片 |
| 发布求助 | 在活动页发"需要帮忙"（搬设备、带食物等） |
| 上传/删除照片 | 管理活动照片集 |

#### 活动状态

| 权限 | 说明 |
|------|------|
| 取消活动 | 取消整场活动，系统自动通知所有已报名者 |
| 公开 ↔ 仅受邀 | 切换活动可见性（公开 → 仅受邀可见，或反向） |
| 转让 Host | 把 Host 身份转给已报名的某人，对方确认后接手全部 Host 权限，原 Host 变为普通参与者 |

#### Host 权限获取与失去

- **获取**：任何已登录成员发起活动时，自动成为该活动的 Host
- **失去**：转让后失去、活动被管理员取消后失去
- **Co-Host**：Host 可指定 1-2 名 Co-Host（通过活动管理页 → 贡献角色 → 添加 Co-Host），共享除"转让"和"取消"以外的所有权限。Co-Host 存储在 `event_roles` 表，`role_label = 'co-host'`。Co-Host 在活动详情页显示为副 Host（头像旁标 co-host 标签）。

### 11.4 权限层级总览

```
管理员（user_role = 'admin'）
  └── 管理后台全部操作
  └── 审核申请（通过/拒绝）
  └── 授权/撤销管理员
  └── 所有 Host 权限（可代操作任意活动）
  └── 所有用户权限

Host（活动发起者，动态角色）
  └── 自己活动的全部管理权限
  └── Co-Host 指定
  └── 转让 Host
  └── 所有用户权限

已登录成员（user_status = 'approved', user_role = 'member'）
  └── 报名、投票、推荐、评论、发起活动、寄感谢卡
  └── 自己内容的编辑/删除/隐私控制
  └── 通知偏好设置

未登录访客
  └── 浏览公开内容（活动列表、电影池、成员墙、社群信息）
  └── 提交加入申请
```

**访客可见内容边界**：

| 页面 | 访客可见 | 不可见 |
|------|---------|--------|
| 动态流 | 公开感谢卡、活动回顾（无照片）、里程碑、电影推荐 | 私密感谢卡、活动照片（缩略图模糊 + "加入后查看"）、话题讨论内容 |
| 活动列表 | 活动名称、Host、日期、剩余席位 | 报名操作（替换为"申请加入后报名"按钮） |
| 电影池 | 电影列表、海报、基本信息、投票数 | 投票操作（替换为"加入后投票"） |
| 成员墙 | 成员数量、头像马赛克 | 具体成员信息（替换为"加入后认识大家"引导） |
| 关于串门儿 | 全部可见 | 无限制 |
| 感谢卡页 | 不可访问 | 底部 Tab 栏未登录时隐藏 |
| 我的页面 | 不可访问 | 底部 Tab 栏未登录时隐藏 |

### 11.5 V1 不做的权限功能

- ❌ 管理员分级（管理员/运营两级权限，团队扩大后再拆）
- ❌ 自定义角色（V1 固定四层：管理员/Host/成员/访客）
- ❌ 操作审计日志（V2 增加敏感操作的日志记录）
- ❌ API 权限管理（V1 无第三方接入）
- ❌ 开放注册或邀请码注册（V1 仅支持申请制）
- ❌ WeChat OAuth 登录（V1 仅 Google + Email magic link）

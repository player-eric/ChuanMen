"我把生产数据库删了"

Day 12，凌晨两点。AI 说"我来帮你重置一下本地数据库"

发生了什么

我让 Claude 帮我推一下 schema 到本地数据库
它跑了 prisma db push --force-reset
我以为操作的是 localhost Docker 里的 Postgres
实际清空的是远程 Render 上的生产数据

↓ 怎么发生的

根本原因：.env 软链接陷阱


// server/prisma.config.ts
import dotenv from 'dotenv';
dotenv.config();   // ← 静默加载 .env 文件

export default defineConfig({
  earlyAccess: true,
  schema: './prisma/schema.prisma',
});

server/.env → 软链接

.env.development = localhost ✅
.env.dev = 远程生产库 ❌

陷阱在于

dotenv.config() 会覆盖
命令行传入的 DATABASE_URL
→ 你以为指定了，其实没用

AI 比人还会"手滑"——但它的手滑是系统性的。人不会连续两次犯同一个错，AI 会。每次遇到类似情况，它都会用同样的方式搞砸。

同一条规则，5 层执行

删库之后，我把"不许碰远程数据库"这条规则，写进了 5 个不同的地方

L1
CLAUDE.md — 写给 AI 的项目说明书
用自然语言告诉 AI：什么能做、什么不能做。相当于给 AI 写了一份入职手册。每次对话开始时自动加载。
L2
Memory — AI 的事故记忆
把这次删库事件写进持久记忆文件，跨会话保留。下次 AI 要碰数据库，会想起"上次这么干出事了"。
L3
settings.json deny 规则 — 硬性拦截
在 Claude Code 配置里禁止执行包含 force-reset 的命令。AI 甚至无法尝试执行这个命令。
L4
guard-db.sh — 运行时检查脚本
每次执行 Prisma 命令前自动跑，检查 .env 软链接指向哪个文件。如果不是 localhost，直接报错退出。
L5
pre-commit hook — 提交时拦截
schema 文件有改动但没有对应 migration SQL 文件？拒绝提交。防止"改了 schema 忘了写迁移"。

实际长什么样

settings.json — deny 规则（L3）


{
  "permissions": {
    "deny": [
      "Bash(npx prisma db push --force-reset:*)",
      "Bash(npx prisma migrate reset:*)",
      "Bash(PRISMA_USER_CONSENT_FOR_DANGEROUS_AI_ACTION:*)"
    ]
  }
}

guard-db.sh — 运行时拦截（L4）


# 检查 .env 软链接指向
if echo "$DB_URL" | grep -q "localhost"; then
  exit 0  # 本地库，放行
fi
echo "🚨 BLOCKED: .env points to REMOTE database!"
echo "   Switch: npm run db:local"
exit 2  # 非零退出 = 拦截命令

Memory 文件（L2）——AI 的"事故报告"，跨会话持久


# feedback_no_force_reset.md
**NEVER run destructive Prisma commands.**
### Incident history:
- 2026-03-17: `prisma db push --force-reset` wiped dev DB
  because prisma.config.ts loads dotenv first.
### Why: .env file wins over shell DATABASE_URL.
AI 不会"记住教训"——你需要把教训变成基础设施。

不是一次做对，是快速试错

353 次提交，近一半是修修补补。这不是质量差——是速度够快。

📧 邮件通知：一个功能改了 11 版

v1 — 有新活动就马上发邮件通知所有人
v2 — 太吵了！成员抱怨收到太多邮件。改成每天汇总一封
v3 — 活动邀请不能等到第二天，改成：活动立即发，其他每日汇总
v4 — 加了"几点发"的配置。默认中午 12 点发
v5 — 发送时间判断逻辑写反了，所有人都收不到。紧急修复
v6-11 — 加生日提醒、热门讨论、活动回顾三个新内容源…继续修
每一版都是用户反馈驱动的。不是"一开始没想到"——是用了才知道该怎么做。

现在：说一句话 → 写完了 → 试一下 → 不对 → 改 → 好了（按分钟算）

就两件事

Vibe coding 能不能跑起来，取决于你能不能搭好这两个底座

安全的 Playground

  • Docker 本地数据库——搞坏了重建
  • .env 软链接切换环境
  • guard-db.sh 自动检查指向
  • 隔离的测试数据库（chuanmen_test）
  • 环境变量优雅降级（S3、邮件、OAuth 都可缺省）
让 AI 放心大胆地试，因为它能搞坏的只有你想让它搞坏的东西。

规范的 CI/CD

  • pre-commit：类型检查 + schema 迁移检查
  • push 前必须 tsc --noEmit + build 通过
  • Additive migration only（只加不删）
  • 170+ 自动化测试覆盖核心 API
  • Schema 变更必须附带 migration SQL
AI 写的代码必须通过和人一样的质量门槛，没有例外。
有了这两个底线，中间怎么 vibe 都行。
playground 负责"不怕搞坏"，CI/CD 负责"搞坏了也上不了线"。

两个实际场景

Playground 救命

Render → Vercel 迁移，8 个 commit：

feat: add Vercel full-stack deployment
fix: reduce serverless function size below 250MB
fix: add SPA catch-all rewrite
fix: add directUrl for Neon pooled connection
fix: make prisma migrate deploy non-fatal

每一个 fix 都是在本地 Docker 里试出来的。
如果直接推生产，每个都是 downtime。

CI 兜底

等位功能 waitlist，pre-commit 拦了两次：

❶ schema 加了 offered 状态但没写 migration SQL → 拒绝提交
❷ tsc 报错：offeredAt 字段类型是 DateTime? 但 service 里没做 null check → 修了才能 commit

AI 写的代码必须过和人一样的质量门槛。
feat: implement complete waitlist system
fix: feed sort — signups bump position; waitlist show correct status

反共识：工具链越简单越好

不需要复杂的 agent 编排。你需要的比你想的少得多。

你不需要的 ❌

  • Skill system / Agent framework
  • 多 agent 协作编排
  • RAG / 向量知识库
  • "Prompt Engineering"
这些都是在解决你还没有的问题。

你需要的 ✅

  • CLAUDE.md — 项目规则，写一次，自动加载
  • 临时 prompt — 当场写，用完即弃
  • Memory — 跨会话持久记忆
prompt 不用"工程"。定义角色、给上下文、说清任务，就行了。

Prompt 不用"工程"

三件事说清楚就行：角色 · 上下文 · 任务

真实例子 —— 我在设计新成员审核流程时的 prompt：

[角色] 你是一个做过 10 年社区运营的资深 PM
[上下文] 我们的新成员流程：申请 → 管理员审核 → 内部公示 3 天 → 自动通过。社区 40 人，每月新增 2-3 人
[任务] 站在申请者的角度，这个流程有哪些让人困惑或焦虑的地方？

又比如这个 PPT 本身：

"帮我 review 一下月度企划这个功能的实现"
→ Claude Code 自动拆成 3 个并行 agent（后端/前端/QA），我没有配置任何东西

真正有价值的不是 prompt 写得多精妙，是你知道该在什么时候问什么问题。
AI 是放大器，不是替代品。放大零还是零。

活例子：刚才准备这个 PPT 的时候

我的 prompt：

"帮我 review 一下月度企划这个功能的实现"

就这一句。没有指定用几个 agent，没有配置任何编排。

Claude Code 自己决定派出 3 个并行 agent：

Backend
schema、service、路由
Frontend
页面、组件、路由
QA
测试覆盖、边界用例
我没有 setup 任何"多 agent 系统"——我只是说了一句话。
编排是工具的事。你的工作是知道什么时候该 review、review 什么。

迭代成本趋近于零之后

开发成本 → 0，意味着什么？每次运营活动都可以随手做个产品。

真实例子——串门儿三月企划：Color Hunting 🎨

成员拍下身边的春天颜色，上传到平台，按色系浏览、点赞、评论。

樱花粉 新叶绿 天空蓝 云白 日落橙 泥土棕

这个"产品"有：独立页面、拍照上传、色系筛选、点赞评论、瀑布流展示、Lightbox 看大图
从提出想法到上线：不到一天。代码已经在跑了。

以前"做个产品"是立项级别的事。现在是"下午有个想法，晚上就上线了"。

但数据不能"用后即焚"

Color Hunting 结束之后：

  • 谁拍了最多照片？谁的作品点赞最多？——参与记录
  • 谁和谁互相给对方点了赞？——互动关系
  • 下个月做"音乐漂流瓶"，还是同一批人——跨企划的社群连接
  • 年底回顾，谁参与了几个企划、最活跃的是谁——成长轨迹
Color Hunting 的页面可以下线。但谁参与了、谁互动了，这些数据要永远留着。

产品变成运营了

实现成本趋近于零之后，价值在哪里？

类比：公众号 / 小红书

任何人都能发内容。发布成本是零。
但真正赚钱的人是谁？

有体系 有目标受众 有市场需求洞察 有转化工具

同样的事正在发生在软件开发上：

过去稀缺的

写代码的能力

实现功能的速度

技术选型的经验

现在稀缺的

需求识别——知道该做什么

数据资产——用户行为和关系

资源资产——社群、渠道、信任

串门儿的例子：

代码 48K 行，开源了，任何人都可以 fork。
但没有人能 fork 走的是：40 个真实成员、29 场已办活动、200+ 张感谢卡、成员之间的互动历史和信任关系。
这些是 git clone 克隆不走的。

需求识别比什么都重要。因为实现已经不是瓶颈了。
壁垒不是代码——是数据资产（用户行为）和资源资产（社群信任）。

前端是皮，后端是骨

如果迭代成本为零，产品架构会长什么样？

前端 = 皮肤 🎨

  • 可以随时变化任何展示形式
  • 每个月度企划长得完全不同
  • 一次性的、情绪化的、有温度的
  • 开发成本趋近于零
对白墙是砖墙风，深夜酒馆是霓虹风，私人书架是木质风——无所谓，随便换。

后端 = 骨骼 🦴

  • 记录谁参与了什么
  • 记录谁和谁产生了连接
  • 跨越每一个临时前端
  • 积累成社区的共同记忆
今天的对白墙数据，明天的深夜酒馆数据，后天的年度报告——都在同一个后端里。

真正的挑战

一次性前端产生的数据，如何保持一致性和整合性？

串门儿正在做的事：

统一的 Submission + Reaction 数据模型
↕
Color Hunting
四月企划
五月企划
…

不管前端长什么样，后端都用同一套 submission/reaction 接口。数据天然互通。

// tests/api/projects.test.ts — 已经在跑的真实测试

  { category: '樱花粉' });

未来产品架构：持久后端 + 一次性前端
后端是河床，前端是水——水的形状一直在变，但河床在那里。

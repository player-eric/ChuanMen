**状态**: 需求确认中

**日期**: 2026-02-21

## 一、核心设计原则

以下 6 条原则贯穿所有功能设计，同等重要，共同约束每一个产品决策。

### 关系优先于活动

串门是一个帮人交朋友的社群，不是活动平台。活动是手段，关系是目的。产品设计始终把"人与人之间的关系"放在活动之上——让关系可见、可追溯、可加温。

### 个性化引导降低参与成本

不是等人自己来，而是在对的时机推一把。通过 AI Agent 理解每个人的状态，在合适的时候用朋友的语气提供个性化引导，而不是群发通知。

### 轻输入，重整理

用户的输入永远是简单的（发一条动态、贴个链接、点个投票）。平台负责把这些输入整理成结构化的内容。用户觉得自己在"发朋友圈"，系统知道这是一条有类型、有关联、可参与的内容。

### 让参与可见

每个人的贡献——推荐、投票、Host、报名、支持——都被系统记录和呈现。不只是"发了就沉了"，而是被整理成清晰的画面：你推荐了多少部电影、参加了多少场活动、有多少人投了你推荐的片子。

### 每条内容都是参与入口

动态流里的每条内容都带有可操作的动作。看到电影推荐 → 投票。看到活动 → 报名。看到提案 → 响应。不是被动浏览，是随时参与。

### 共创而非分发

所有面向成员的内容来自成员自身的贡献，不是运营单方面的输出。平台的价值来自所有成员的参与叠加，而非中心化的内容生产。

## 二、产品定位

### 一句话

串门网页端是一个让社群成员共同创造内容、共同决定做什么、共同记录经历的平台。

### 终极形态

一个基于共同价值观的私域社交平台。前台是轻松的动态流，人人随手参与；后台是结构化的社群知识库，把所有人的参与整理成清晰的集体成果。每个人都能看到自己在集体中的位置。

### V1 要验证的核心假设

> **"让参与可见"这件事，真的能让人更想参与吗？**

### 与现有工具的关系

- **微信群**：继续承担日常聊天，网页端不替代即时通讯
- **Notion**：历史数据来源，一次性迁移后退役为运营内容编辑工具（串门原则、Host 指南等）
- **小红书/Luma**：继续承担对外拉新，网页端服务已在社群内的人

## 三、两个核心循环

串门网页端的功能围绕两个循环展开，它们在"活动"节点交汇，在"人"的维度沉淀。

### 循环1：推荐 → 响应 → 选中 → 完成

一个人的品味或想法，经过集体的响应和筛选，变成所有人共享的体验。

```
某人推荐了一部电影
  → 其他人投票"我也想看"
    → Host 办电影夜时选中了它
      → 大家一起看了，变成共同记忆
```

**V1 载体：电影推荐池、活动提案**

### 循环2：发起 → 报名 → 参与 → 记录

一个人愿意打开门，一群人走进来，一段经历被记录下来。

```
某人发起一场活动
  → 成员报名
    → 线下一起做了
      → 照片、参与者、内容被记录
```

**V1 载体：活动日历、活动记录**

### 交汇与沉淀

两个循环在"活动"节点交汇——循环1 产出"我们想做什么"，循环2 产出"我们做了什么"。而每次交汇都沉淀出人与人之间的关系数据：共同品味、共同经历、互相感谢。这些关系数据通过"人与人连接层"（见 5.5）反哺下一次循环。

### 4.1 首页（动态流）

**定位**：社群的心跳。聚合所有模块的更新，让人每天想来看看。

**内容来源**（从各模块聚合）：

- 🎬 有人推荐了一部新电影
- 👍 一部电影获得了新投票
- 💡 有人提了一个新活动提案
- 📅 有人发起了一场新活动
- 📸 一场活动刚结束，有了回顾/照片
- ✉ 有人寄了一张公开感谢卡
- 🌱 有人的种子计划有了新进展
- 💬 有人发起了一个话题讨论
- 🎲 本周小局抽签结果 / 小局完成记录
- 🎉 集体里程碑（见 5.5 集体里程碑）

**视觉层级**：

动态流不是所有卡片等高等宽。三层视觉权重让用户在滚动时有节奏感：

| 层级 | 适用内容 | 视觉特征 |
|------|---------|---------|
| Tier 1 · 高亮 | Agent 横幅、集体里程碑 | 全宽、边框发光、最醒目 |
| Tier 2 · 标准 | 活动、活动回顾、感谢卡、讨论、种子更新 | Card 组件包裹，标准内边距 |
| Tier 3 · 紧凑 | 电影推荐、活动提案 | 无 Card 包裹，s1 底色 + 细边框，更矮。同类内容聚合为一组（如"🎬 新片推荐"） |

**时间分隔线**：动态按时间倒序，用"今天"、"本周"、"更早"分隔线划分区段。分隔线样式为左右细线 + 居中文字，视觉极轻。

**点击穿透**：Feed 中的活动卡片、电影推荐、提案均可点击跳转到对应详情页面（活动→活动页，电影→推荐页，提案→活动页提案 tab）。投票/报名等按钮有 stopPropagation，不会误触跳转。

**头像穿透**：全站所有页面中出现的头像和用户名均可点击，跳转到该成员的详情页（与我的共同经历）。这是进入他人 profile 的主要入口，不需要绕道成员墙。动态流、活动详情、感谢卡、讨论回复中的头像和名字均适用。

**空状态（新用户首次打开）**：

新用户看到的不是空白页面，而是一个欢迎引导：
- Agent 欢迎横幅：介绍串门儿是什么
- 四个快捷入口：看看活动 / 推荐电影 / 了解串门 / 寄感谢卡
- 底部提示："动态流会随着你参加活动慢慢丰富起来 ✨"

**每条动态的结构**：

```
[头像] [用户名] [动作描述] [时间]
[内容卡片（根据类型不同呈现不同样式）]
[参与动作按钮] [点赞] [评论]
```

**参与动作（因内容类型而异）**：

| 内容类型 | 参与动作 |
|---------|---------|
| 电影推荐 | "我也想看"投票 |
| 活动提案 | "感兴趣"投票 |
| 活动发起 | 报名 / Waitlist |
| 活动回顾 | 点赞、评论 |
| 公开感谢卡 | 点赞 |
| 种子计划更新 | 支持、评论、参与协作 |
| 话题讨论 | 回复、点赞 |

**通用动作**：点赞、评论（所有内容类型都支持）

**排序逻辑**：按时间倒序，V1 不做算法推荐。Agent 生成的社群级内容（集体里程碑、Host 致敬、供给求助）可置顶。

**V1 数据来源**：从各数据表聚合查询生成动态卡片（见第六章）

### 4.2 电影池

**定位**：社群共同积累的电影品味库。位于「👍 推荐」Tab，电影是 V1 唯一品类。未来可扩展菜谱、音乐、好店、游戏等品类（品类横滑选择器已预留）。

**浏览视图**：

- 所有电影列表，显示：封面、片名、年份、豆瓣评分、推荐人、投票数、状态（候选/已放映）
- 排序：默认按投票数降序，可切换为最新推荐、最近放映
- 筛选：候选 / 已放映 / 全部

**用户动作**：

| 动作 | 说明 | 需要登录 |
|------|------|---------|
| 浏览 | 查看所有电影及详情 | 否 |
| 投票"我想看" | 对任意候选电影投票，投票可取消 | 是 |
| 推荐新电影 | 贴豆瓣链接，系统自动抓取信息（片名、封面、导演、年份、评分），加入池子 | 是 |
| 提名到活动 | 把电影推荐到某次电影夜（关联活动） | 是 |

**电影详情页**：

- 电影基本信息（封面、导演、年份、评分、简介）
- 推荐人
- 投票人列表 + 总票数
- 关联活动（被提名到哪些活动 / 在哪次活动放映过）
- 放映记录（日期、Host、地点）

**Host 选片视图**（在活动页面内）：

对每部被提名的电影显示：
- 总投票数（社群整体偏好）
- 本场报名者投票数（参加这次活动的人中多少人想看）

**状态流转**：

```
候选（在池子里）→ 已放映（被某次活动选为放映片，关联放映场次）
```

### 4.3 活动提案

**定位**：任何人都能提出想法，社群投票响应，有人认领执行。位于「📅 活动」Tab 的「提案」子页签。

**统一创建入口**：活动页底部有一个"+"浮动按钮（FAB），点击展开两个选项：
- **🏠 发起活动**：直接创建正式活动（进入发起流程）
- **💡 提一个想法**：发起提案，等待社群响应

**浏览视图**：

- 所有提案列表，显示：标题、提出者、投票数、状态（讨论中/已排期/已完成/取消）
- 排序：默认讨论中的在前，按投票数降序

**用户动作**：

| 动作 | 说明 | 需要登录 |
|------|------|---------|
| 浏览 | 查看所有提案 | 否 |
| 投票"感兴趣" | 对任意讨论中的提案投票 | 是 |
| 提交新提案 | 写标题 + 描述，发起一个新想法 | 是 |
| 🏠 我来组织 | 认领一个提案，转为正式活动 | 是 |
| 评论 | 在提案下方讨论 | 是 |

**想法→活动转化**：提案卡片上同时显示"🙋 感兴趣"和"🏠 我来组织"两个按钮。任何人点击"我来组织"后，系统创建一个预填了提案信息的活动草稿，组织者补充时间、地点、人数后发布。原提案状态自动流转为"已排期"。

**状态流转**：

```
讨论中 → 已排期（有人认领，创建了对应活动）
讨论中 → 已完成
讨论中 → 取消
```

**与活动的关联**：提案被认领后，创建对应的活动日历条目，两者关联。

### 4.4 活动日历

**定位**：接下来要发生什么，以及报名参加。位于「📅 活动」Tab 的「即将到来」子页签。

**关键约束**：活动建议人数上限不超过 10 人。小规模是串门的核心体验，保持亲密感。

#### 本周小局（抽签做东机制）

活动页「即将到来」顶部固定一张"本周小局"卡片，展示本周抽签结果。

**机制**：
- 系统每周从活跃成员中随机抽签一人，作为本周"做东人"
- 做东人收到通知："轮到你了！"
- 约谁、做什么、几个人，完全由做东人决定
- 做东人点击"发起小局"→ 进入轻量版活动创建流程（预填 Tag 为"小局"）
- 见面后简单记录 → 出现在 feed

**解决的问题**：给"谁都不好意思先开口"一个正当理由——不是我主动约你，是我被抽中了必须约人。

**卡片展示**：
- 🎲 图标 + "本周小局" 标题 + 期数
- 做东人头像和名字
- "轮到你做东了！"提示
- 两个按钮：发起小局 / 历史记录

**数据**：抽签记录（期数、被抽中人、是否完成、关联活动 ID）

**浏览视图**：

- 日历视图（月视图）
- 列表视图（按时间排序的活动卡片）
- 每张活动卡片显示：活动名称、日期时间、Tag（电影/串门儿/节日/徒步/户外）、Host、剩余名额

**活动详情页**：

- 活动基本信息（名称、日期、地点、Host、人数限制）
- Tag 分类
- 报名者列表（显示受邀 / 已报名状态）
- **社交上下文提示**（见 5.5）："XX 也报名了（你们一起参加过 3 次活动）"
- Waitlist
- 活动记录人
- 候选电影列表（如果是电影夜）+ Host 选片视图
- 放映电影（确定后显示）
- 活动内容/说明（Host 写的 house rules 等）
- Host 求助区域（如需要人帮忙搬设备、带食物等）

#### 活动发起流程（两阶段邀请机制）

活动发布不是一次性广播，而是分阶段释放名额，让 Host 对参与者有控制感：

**第一阶段：Host 创建 + 定向邀请**

```
Host 创建活动
  → 系统引导：填写名称、日期、地点、人数上限（建议≤10）、Tag、说明
  → 系统引导："想先邀请谁？"
    → 系统提供建议（见 5.5 社交上下文）：
      "以下成员参加过你上次的活动且待到了最后：XX、YY、ZZ"
      "以下新成员还没参加过你的活动类型，可以考虑邀请"
  → Host 从成员列表选择想定向邀请的人
  → 被邀请人收到私人通知："XX 邀请你参加 XX 活动"
  → 被邀请人可接受 / 拒绝 / 暂不回复
```

此阶段活动仅对被邀请人可见。Host 可以先锁定自己最想邀请的核心参与者。

**Host 可见性控制**：Host 在创建活动时可以设置"不对以下成员显示"名单。被排除的成员看不到这场活动的存在——不是"报名被拒"，而是根本不知道有这场活动。比"拒绝"更体面，保护双方面子。此设置仅 Host 和运营可见。

**第二阶段：公开广播（活动前约一周，具体时间待定）**

```
定向邀请窗口结束
  → 系统自动将活动发布到动态流 + 活动日历
  → 对所有可见成员公开
  → 剩余名额开放报名（先到先得）
  → 满员后自动切换为 Waitlist 模式
  → 公开阶段 Host 仍可审核报名，接受或拒绝
```

**参与前引导（首次报名触发）**

当用户第一次报名某场活动时，系统在确认报名前插入一个轻量引导页：

```
报名确认前展示：
  1. 串门原则摘要（3-5 条核心要点，不是全文）
     - 真诚是前提
     - 对 Host 礼貌（准时、遵守 house rules、帮忙收拾）
     - 不推销、不越界
  2. 本场活动的 house rules（Host 写的注意事项）
  3. "我已了解" → 确认报名
```

非首次参加的用户跳过原则摘要，只展示本场 house rules（如果 Host 写了的话）。

**Waitlist 机制**：

满员后用户可加入 Waitlist。Waitlist 同时是社群供给不足的信号——Agent 会根据 Waitlist 情况触发 Host 挖掘引导（见 5.4 Agent 目标 D）。

**用户动作**：

| 动作 | 说明 | 需要登录 |
|------|------|---------|
| 浏览 | 查看所有公开活动 | 否 |
| 报名 | 参加活动（公开阶段，有名额时） | 是 |
| 接受邀请 | 响应 Host 的定向邀请 | 是 |
| 加入 Waitlist | 活动满员时排队 | 是 |
| 取消报名 | 退出已报名的活动 | 是 |
| 发起活动 | 创建新活动（进入发起流程） | 是 |
| 提名电影 | 对电影夜活动提名候选电影 | 是 |
| 选片 | Host 选定放映电影 | 是（Host 权限） |
| 管理报名 | 查看/接受/拒绝报名 | 是（Host 权限） |

#### 小局（PopupHost 乱盘）

**本质**：解决"谁都不好意思先开口约人"的问题。系统每周抽签指定一个人做东，给 TA 一个正当理由去主动约人。

**核心流程**：

```
系统每周抽签 → 抽中一个人（潜在 Host）
  → 被抽中的人收到通知："🎲 轮到你做东了！"
  → 约谁、做什么、几个人，全由自己决定
  → 被抽中的人发起一个小局（走轻量版发起活动流程）
  → 见面后简单记录 → 出现在动态流
```

**与普通活动的区别**：普通活动是"有人主动发起"，小局是"机制指定一个人，给 TA 发起的理由"。不是配对，不是强制，是降低开口门槛。

**页面呈现**：

活动页「即将到来」tab 顶部置顶一张特殊卡片——「🎲 本周小局」：
- 显示被抽中的人（头像、名字）
- 文案："轮到你做东了！约谁、做什么、几个人，你说了算。"
- 主按钮："🏠 发起小局"→ 进入轻量版发起活动流程（预填 Tag 为"小局"）
- 次按钮："历史"→ 查看过往小局记录

**抽签规则**：
- 每周一自动抽签，从活跃成员中随机选一人
- 最近 4 周内已被抽中的人不重复
- 从未 Host 过的人优先（更高权重）——小局是零压力的 Host 入门体验
- 被抽中的人可以"跳过"（本周不方便），系统重新抽
- 连续 2 周跳过的人降低被抽中的概率

**与 Host 培养的关系**：小局是最轻量的 Host 体验。约两三个人喝杯咖啡，就完成了一次"做东"。Agent 可以在小局结束后引导："上次小局玩得开心吗？要不要试试在家办一次电影夜？"——从小局自然过渡到正式 Host。

### 5.4 运营 AI Agent

这是整个平台最重要的运营杠杆。不是等人自己来，而是在对的时机推一把。不是写 30 条 if-then 规则，而是让 AI 理解社群的原则和目标，自主判断该对谁、在什么时候、说什么。

#### 为什么用 Agent 而不是规则引擎

规则引擎的问题：
- 规则只会越来越多，每条都是硬编码的 if-then，无法处理组合情况和边界情况
- 文案千篇一律，无法个性化
- 维护成本随规则数量线性增长

AI Agent 的优势：
- 规则变成 context（Agent 的 playbook），而不是代码
- Agent 能综合多个信号做判断（这个人最近参与频率上升 + 刚在提案投了票 + 上次活动待到最后 = 现在是邀请 co-host 的好时机）
- 文案可以根据每个人的情况个性化
- 新增"规则"只需要更新 playbook 的文字描述，不需要改代码

#### Agent 架构

```
┌─────────────────────────────────────────┐
│            运营 AI Agent                 │
│                                         │
│  输入：                                  │
│  ├── 社群数据快照（从数据库读取）        │
│  │   ├── 所有成员及其参与历史              │
│  │   ├── 所有活动（过去 + 即将到来）        │
│  │   ├── 电影投票、提案、种子计划等         │
│  │   └── 上次推送记录（避免重复）           │
│  │                                       │
│  ├── 运营 Playbook（Agent 的 system prompt）│
│  │   ├── 串门原则和价值观                  │
│  │   ├── 7 大引导目标（见下文）            │
│  │   ├── 频率约束和礼仪规则               │
│  │   └── 文风指南（串门儿的语气）          │
│  │                                       │
│  └── 今日事件（新活动、活动结束、新报名等）  │
│                                         │
│  输出：                                  │
│  ├── 个人推送列表                         │
│  │   每条包含：收件人、内容、行动链接、渠道  │
│  ├── 社群级动态（动态流置顶卡片）           │
│  └── 运营日报（给管理员的诊断摘要）         │
└─────────────────────────────────────────┘
```

#### 运行机制

```
每天运行 2 次（早 9 点、晚 7 点）：

1. 读取数据库全量快照
2. 读取上次运行以来的变更事件
3. Agent 结合 Playbook + 数据 + 事件，为每个成员生成推送建议
4. 约束检查（见"频率约束"）
5. 通过 Email 发送（附带行动按钮链接）
6. 网页端登录后展示相关引导卡片
7. 输出运营日报给管理员
```

**V1 关键决策：Agent 输出先经过人工审核。** 初期至少前 2 周，Agent 生成的推送由运营确认后再发送。确认质量稳定后切换为自动发送 + 运营抽检。

#### Agent Playbook：7 大引导目标

以下不是 if-then 规则，而是 Agent 的目标描述和判断参考。Agent 根据这些目标，结合每个用户的实际情况，自主决定推送什么。

**目标 A：活动参与引导**

让成员知道有什么活动可以参加，降低参与门槛。

关注的信号：
- 新活动进入公开阶段 → 通知可能感兴趣的成员（基于活动类型和参与历史）
- Host 定向邀请 → 立即通知被邀请人（最高优先级）
- 活动 24h 前 → 提醒已报名的人，附带 house rules
- 用户投票过的电影被选为放映片 → 精准通知
- Waitlist 递补 → 立即通知

Agent 个性化判断示例：
- "XX 参加过 3 次电影夜但从没参加过户外活动，这次有个徒步刚好在她家附近，值得推"
- "YY 上周刚加入 Waitlist 被放了鸽子，这周有同类型活动应该优先通知"

**目标 B：投票与推荐引导**

让成员参与内容共建，降低贡献门槛。

关注的信号：
- 电影夜有新提名 → 通知报名了该电影夜的人去投票
- 每周新片汇总 → 活跃投票者收到摘要
- 提案有票但无人认领 → 通知有能力认领的人

**目标 C：活动后沉淀与连接**

活动结束后是情感最高点，抓住这个时刻促进记录和人际连接。

关注的信号：
- 活动结束当天/次日 → 引导参与者上传照片
- 活动结束后 → 引导给 Host 发感谢卡（见 5.5）
- 活动结束后 → 引导给今天印象深刻的人发一句话（见 5.5）
- 记录人提醒

**目标 D：Host 挖掘与培养**

保证社群的活动供给。这是最重要的目标。

Agent 应该综合判断潜在 Host，不是机械触发。比如一个人虽然参加了 3 次但每次都提前走，可能不是好的 Host 候选人。而一个只参加了 2 次但每次都待到最后帮忙收拾的人，可能更合适。

三个层次：

**个人层**——识别潜在 Host 并在合适时机推送：
- Waitlist 触发（最强信号）："活动满了，要不要自己组一场？"
- 活跃但从未 Host → 建议 co-host，并推荐一个有经验的搭档（见 5.6 Co-Host 配对）
- 刚参加完活动的情绪高点 → 轻推
- 有经验的 Host 超过 3 周没办活动 → 温和提醒
- 首次 Host 了某个小众类型 → 引导变为定期活动
- 多次参加某类型但从未 Host → "你这么喜欢 XX，要不要自己组一次？"

**社群层**——供给不足时公开求助：
- 本周没有活动 → 动态流置顶求助卡片
- 连续 2 周活动 < 2 场 → 社群级广播
- Waitlist 比例过高 → "社群需要更多 Host"
- 某类型活动长期空缺 → 针对性求助

**Host 求助**——Host 遇到困难时社群帮忙：
- 报名人数不够 → 动态流补推
- 需要帮忙（搬设备、带零食） → 活动详情页"Host 求助"区域
- 首次 Host 紧张 → 自动配对 co-host 搭档

**目标 E：新人融入**

让新人快速找到归属感，关键前 3 次体验。

- 首次注册 → 引导阅读串门原则
- 首次报名 → 报名确认前展示原则摘要 + house rules（见 4.4）
- 首次活动结束 → 引导浏览更多活动 + 引导感谢 Host + 引导给印象深刻的人发消息
- 7 天未参加活动 → 推荐近期活动

Agent 对新人应该更主动、更温暖，但不轰炸。

**目标 F：邀请增长**

鼓励成员带朋友来，但保持低压力。

- 活动后 → "有朋友也会喜欢吗？"
- 成员满 1 个月 → "有想推荐的人吗？"
- Host 发起活动 → "也邀请你的朋友来！"

**目标 G：再激活**

拉回沉默用户，但要体面。

- 3 周未参与 → "好久没见你了"+ 近期活动
- 种子计划 2 周没更新 → 温和提醒
- 如果一个人连续被推送 2 次都没反应，降低频率而不是加大力度

#### Agent 文风指南

Agent 生成的推送文案应符合串门儿的语气：
- 像朋友说话，不像平台通知
- 可以带一点自嘲和幽默
- 避免营销感、紧迫感制造（"最后 2 个名额！"这种不要）
- 参考串门儿内容风格指南的"Mean + 懦弱 + 搞笑 + 真诚"

好的例子：
- "你投的《花样年华》被选中了，这周六在白开水家放映，要不要来？说实话我觉得你不来挺亏的"
- "上次你在 XX 家的电影夜，大家聊到快 12 点，好像很开心？要不要试试自己也办一次？不用完美，3-4 个人就够了"

不好的例子：
- "🔥 热门活动即将开始！名额有限，立即报名！"
- "尊敬的用户，我们注意到您最近未参与活动..."

#### 运营日报（给管理员）

Agent 每天额外输出一份运营摘要：

```
串门运营日报 2026-02-10

📊 社群状态
- 本周活动数：3（正常）
- 活跃 Host：4 人（较上周 -1）
- 本周新人：2 人（XX、YY）
- Waitlist 比例：25%（健康）

🎯 今日推送
- 发送了 15 条个人推送（活动提醒 8、Host 引导 3、新人融入 2、感谢引导 2）
- 发送了 1 条社群级动态（电影夜提案征集）

💡 运营建议
- XX 参加了 4 次活动，每次都待到最后，建议私聊邀请 co-host
- 徒步类活动已经 3 周没有了，但提案里有 5 人投票想要，建议找人组织
- YY 是新人，第一次活动体验不错（待到了最后），建议下次活动主动邀请

⚠️ 关注
- ZZ 连续 4 周没参加活动，之前很活跃，可能需要人工关心一下
```

Agent 处理模式识别，人处理关系关怀。

#### 推送渠道

| 渠道 | 用途 | V1 实现 |
|------|------|---------|
| **Email** | 个人推送主渠道 | Agent 输出 → Email 发送 |
| **网页端卡片** | 登录后显示相关引导 | 根据 Agent 输出渲染 |
| **动态流置顶** | 社群级公告和求助 | Agent 输出 → 自动生成置顶卡片 |

未来方向：App push notification、站内通知中心

#### 频率约束（Agent 必须遵守）

- 每人每天最多 2 条推送
- 同一目标对同一用户 7 天内不重复
- Host 挖掘类每人每月最多 2 次
- 用户已关闭的推送类型跳过
- 连续 2 次推送无反应 → 自动降频
- 新人前 2 周可适当放宽（每天最多 3 条），之后回归正常

### 5.5 人与人的连接层

这是设计原则「关系优先于活动」的核心实现。三个机制，从被动到主动：

#### 共同品味与共同记忆（被动展示）

当用户访问另一个成员的个人页面时，系统自动展示两人之间的关联：

```
你和 XX 的共同经历：

🎬 共同品味
  - 你们都投了：《花样年华》《惊魂记》《永恒和一日》
  - 你们都喜欢：文艺片、日本电影

📅 一起参加的活动
  - 12/14 白开水家电影夜
  - 1/5 新年 Potluck
  - 1/20 Central Park 徒步

🌱 共同关注的种子计划
  - XX 的「学日语」项目（你点过赞）
```

实现方式：纯数据交叉计算，不需要额外用户输入。**这不是推荐算法，是事实展示。**它把模糊的"我好像在哪见过你"变成具体的共同记忆。

#### 感谢卡（主动连接）

活动结束后，Agent 引导参与者发送感谢卡。感谢卡是串门的核心社交货币——比点赞重，比私信轻。

**隐私模式**：

| 模式 | 说明 | 展示位置 |
|------|------|---------|
| 🌐 公开 | 所有人可见 | 动态流 + 收件人个人页面 + 感谢卡页 |
| 🔒 私密 | 仅收件人可见 | 收件人个人页面（带 🔒 标记） |

发送时默认私密，发送者可选择公开。

**给 Host 的感谢卡**：
- 提供预设标签一键选择："好吃！""氛围很棒""下次还来""Host 太用心了"
- 可选：展开写一句话
- 可选：附上活动场景照片（系统提供渐变氛围图兜底）
- Host 收到通知 + 感谢卡展示在个人页面"收到的感谢卡"区域

**给同场参与者的感谢卡**（更重要）：
- Agent 推送引导："今天谁让你印象最深？给 TA 说句话"
- 选一个人 → 写一句话或选预设标签："聊得很开心""很有趣的人""下次还想见到你"
- 默认私密，仅收件人能看到
- 收件人收到通知："XX 在「XX 活动」后给你寄了一张感谢卡"

**感谢卡页**（✉ Tab）：
- **单页布局，无 Tab 切换**
- 顶部余额条：本月配额（如"本月还能寄 2/4 张 · 月底清零"）
- 寄卡流程：三步进度条（选人 → 写话+拍照+选可见性 → 预览寄出）
- 收到的感谢卡：标题"收到的感谢卡 · X 张"，支持展开/叠放视图切换
- **空状态**：新用户看到引导文案"参加一次活动后，就可以给同行的人寄一张感谢卡"+ 跳转活动页按钮
- **配额机制**：每人每月 4 张感谢卡，月底清零。配额制造稀缺感，让每张卡更有重量。

为什么重要：
- 它把"我们一起参加了一个活动"变成"你专门对我说了一句话"
- 私密模式让人更愿意表达真实感受
- 公开模式让社群看见感恩的文化，鼓励更多人表达
- 它创造了活动之外的人际信号，是关系加温的起点
- 数据层面：谁给谁发过感谢卡 = 一种隐性的关系强度指标
- 称号系统的输入源：感谢卡的预设标签就是称号的"选票"（见 5.6）

#### 社交上下文提示（活动报名与邀请时）

**报名时**——当用户查看活动详情页时，展示社交上下文：

```
XX 也报名了（你们一起参加过 3 次活动）
YY 也报名了（你们都投过《花样年华》）
还有 3 个你还没见过的新面孔
```

这把报名决策从"这个活动有意思吗"变成"我想见的人在不在"。

**Host 邀请时**——Host 创建活动选择定向邀请时，系统提供建议：
- "以下成员参加过你上次的活动且待到了最后：XX、YY、ZZ"
- "以下新成员还没参加过你的活动类型，可以考虑邀请"

#### 集体里程碑（社群归属感）

在动态流中穿插社群整体的成就，让人感受到"我是一个正在成长的集体的一部分"：

- "串门第 50 场活动 🎉 从去年 10 月到现在，我们一起看了 XX 部电影、吃了 XX 顿饭、走了 XX 次路"
- "电影池突破 100 部！最受欢迎的 3 部是..."
- "本月 8 位 Host 开放了自己的客厅，感谢 XX、YY、ZZ..."
- "XX 和 YY 从电影夜认识，现在成了室友"（需要当事人同意）

由 Agent 在运营日报中建议，运营确认后发布。

#### 关系数据的使用原则

- **所有关系信号都是私密的**：谁给谁发了私密感谢卡，只有当事人知道
- **系统展示的是事实，不是判断**：显示"你们一起参加过 3 次活动"，不显示"你们关系很好"
- **不做社交图谱排名**：不展示"最受欢迎的人"、"连接最多的人"，避免社交压力
- **服务于线下连接**：所有线上的关系信号，目标都是促成下一次线下见面

### 5.6 Host 激励体系

好的 Host 是社群的发动机。用一切手段让 Host 感到被看见、被感谢、被回馈。

**核心理念**："贡献的人不花钱，搭便车的人付钱。"

#### Host 可见性

Host 的贡献在平台上被充分展示：
- **成员墙 & 个人页面**：显示 Host 次数、Host 过的活动列表
- **活动记录**：每场活动的 Host 被突出展示
- **动态流**：Host 发起活动时，动态卡片突出 Host 身份

#### 荣誉体系

| 机制 | 说明 | V1 是否实现 |
|------|------|------------|
| **Host 徽章** | 个人头像旁显示 Host 标识（如小房子图标） | ✅ V1 |
| **Host 次数里程碑** | 首次 Host 🏠、5 次 ⭐、10 次 🔥、20 次 👑 | ✅ V1 |
| **月度 Host 致敬** | 每月动态流置顶"本月感谢这些 Host"（Agent 建议，运营确认） | ✅ V1 |
| **年度 Host 致敬** | 年度感谢（侧重"感谢这些人"而非排名竞争） | ⚠️ 谨慎 |

#### 称号与身份系统

称号不是自己给的，是别人给的。主打亲切好玩，不搞稀缺。

**社群称号（来自感谢卡积累）**

感谢卡（见 5.5）的预设标签就是称号的"选票"。累积收到同类标签达到一定次数，自动获得对应称号，显示在个人页面和成员墙头像旁。

| 标签方向 | 对应称号举例 | 触发条件（参考） |
|---------|------------|----------------|
| "好吃！""做饭太厉害了" | 🍳 厨神 | 累计收到 3 次 |
| "选片好""品味太好了" | 🎬 选片人 | 推荐的电影被放映 3 次 |
| "氛围很棒""很会聊天" | 🔥 氛围担当 | 累计收到 3 次 |
| "第一次来就很放松" | 🧊 破冰王 | 有新人的活动中累计收到 3 次 |
| "每次都帮忙收拾" | 🧹 收拾之神 | 累计收到 3 次 |
| "拍照太好了" | 📸 首席摄影 | 累计收到 3 次 |

- 一个人可以同时拥有多个称号
- 称号永久保留，越攒越多
- 阈值故意设低，让称号容易获得——重点是好玩和被看见，不是稀缺
- 新称号获得时在动态流生成一条卡片："XX 被大家封为「厨神」🍳"

**运营身份（运营指定）**

社群志愿者的职责标识，由运营在后台指定，显示在个人页面和成员墙。

| 身份 | 说明 |
|------|------|
| 🎨 设计总管 | 负责海报、视觉 |
| 📱 社媒小助手 | 负责小红书等对外内容 |
| 📷 活动摄影师 | 负责活动记录拍摄 |
| ✍️ 内容主笔 | 负责 Newsletter / 活动回顾 |
| 🔧 技术支持 | 负责平台相关 |

运营身份可随时调整，不是永久的。有运营身份的人同时也可以拥有社群称号。

#### 实质回馈

| 机制 | 说明 | V1 是否实现 |
|------|------|------------|
| **会员费减免** | Host 达到一定次数，会员费抵扣或免除 | 🔜 会员制上线后 |
| **贡献者免费** | 活跃 Host 自动进入"贡献者"身份，不需要付费 | 🔜 会员制上线后 |
| **优先报名** | Host 报名其他人的活动时享有优先权 | ✅ V1 |
| **专属活动** | Host-only 的交流活动（互相学习、吐槽、支持） | 线下运营，平台记录 |

#### Host 培养路径

系统不仅激励现有 Host，还要系统性地把参与者培养成 Host。Agent（目标 D）负责识别时机和推送，以下是完整路径：

```
参与者 → 活跃参与者 → Co-Host → 独立 Host → 多类型 Host → 资深 Host

  参加 1 次：了解串门是什么
  参加 3 次：Agent 建议 co-host + 自动推荐搭档（见下方 Co-Host 配对）
  Co-Host 1 次：Agent 建议独立 Host
  独立 Host 1 次：获得 Host 徽章 🏠
  首次 Host 某小众类型：Agent 引导"要不要变成定期活动？"
  Host 5 次：里程碑 ⭐ + 会员费减免
  Host 10+ 次：资深 Host 🔥 + 贡献者身份（免费）
```

**Co-Host 自动配对机制**：

当 Agent 识别到潜在 Host 时，不只推送"要不要试试"，而是主动匹配：
- 从活跃 Host 中选一个有经验的人
- 同时向双方推送："XX 想试试 co-host，YY 你愿意带 TA 一起办一次吗？"
- 双方同意后进入 co-host 模式

#### Host 保护

Host 愿意打开家门，平台必须保护这种信任：

- **静默排除**：创建活动时可设置"不可见名单"（见 4.4）
- **定向邀请优先**：两阶段机制让 Host 先锁定想邀请的人（见 4.4）
- **报名审核权**：公开阶段 Host 仍可审核报名
- **多次拒绝上报**：一个人被多个 Host 拒绝时，系统标记并通知运营介入
- **Host 求助不丢人**：活动详情页的"Host 求助"区域让协作成为常态
- **活动后匿名反馈**（V2）：让 Host 知道哪里可以改进，但不公开

## 六、数据架构

### 总体策略

V1 使用独立数据库（PostgreSQL / Supabase），不依赖 Notion API 作为运行时数据源。Notion 退役为运营后台备份和内容编辑工具。

**为什么不继续用 Notion 做数据源**：
- Notion API 读写速度慢，不适合实时交互
- 数据模型受限于 Notion 的表结构，难以表达关系型数据
- 感谢卡、话题讨论、协作者等新功能无法用 Notion 建模
- 独立数据库支持全文搜索、关系查询、实时订阅

### 核心数据表

| 数据表 | 说明 | 关键字段 |
|--------|------|---------|
| users | 成员 | id, name, email, avatar, bio, role, joined_at |
| events | 活动 | id, title, host_id, date, location, capacity, phase(invite/open), tags |
| event_signups | 报名 | event_id, user_id, status(invited/accepted/waitlist), invited_by |
| movies | 电影 | id, title, year, director, poster, douban_url, recommended_by, status |
| movie_votes | 投票 | movie_id, user_id, created_at |
| proposals | 提案 | id, title, author_id, description, status, created_at |
| proposal_votes | 提案投票 | proposal_id, user_id |
| postcards | 感谢卡 | id, from_id, to_id, message, tags[], event_id, visibility(public/private), photo, created_at |
| seeds | 种子计划 | id, title, author_id, description, status |
| seed_collaborators | 协作者 | seed_id, user_id, role_label |
| seed_updates | 进度更新 | seed_id, content, created_at |
| discussions | 话题讨论 | id, author_id, title, body, created_at |
| discussion_replies | 回复 | discussion_id, author_id, content, created_at |
| about_content | 关于串门儿 | id, type(principle/host_guide/letter/about), title, content_md, updated_at |
| weekly_lottery | 每周小局抽签 | id, week_number, drawn_member_id, status(pending/accepted/completed/skipped), event_id(关联活动), created_at |
| experiment_pairings | 配对 | experiment_id, user_a_id, user_b_id, met_at |
| agent_pushes | Agent 推送 | id, user_id, goal, content, channel, sent_at, opened, acted |
| user_preferences | 用户设置 | user_id, push_email, push_frequency, etc. |
| likes | 点赞 | entity_type, entity_id, user_id |
| comments | 评论 | entity_type, entity_id, author_id, content, created_at |

### 动态流生成

动态流不存储为独立表，而是运行时从各数据表聚合生成。聚合逻辑：

```
SELECT * FROM (
  SELECT 'movie' as type, ... FROM movies WHERE created_at > ?
  UNION ALL
  SELECT 'event' as type, ... FROM events WHERE created_at > ?
  UNION ALL
  SELECT 'postcard' as type, ... FROM postcards WHERE visibility = 'public'
  UNION ALL
  SELECT 'discussion' as type, ... FROM discussions
  UNION ALL
  ...
) ORDER BY created_at DESC
```

Agent 生成的置顶内容（里程碑、求助）通过 agent_pushes 表标记为 feed 类型。

### Notion 数据迁移

V1 启动前一次性从 Notion 导入历史数据（成员、电影、活动记录），之后 Notion 不再作为数据源。运营团队如需在 Notion 编辑内容（如串门原则、Host 指南），通过定时同步或手动触发将变更同步到网页端数据库。

### 认证

- Email magic link（V1）
- 用户表通过 email 匹配身份

## 七、V1 不做的事情

明确排除，避免范围蔓延：

- ❌ 即时通讯 / 群聊（继续用微信）
- ❌ 用户在网页端发布自由形态的动态（V1 动态流从数据库聚合生成，话题讨论是唯一的用户主动发布类型）
- ❌ 智能内容识别（贴链接自动分类为电影/餐厅等，V2 方向）
- ❌ 多社群/多城市支持（V1 服务串门儿一个社群）
- ❌ 算法推荐排序（V1 按时间倒序）
- ❌ 站内通知中心（V1 通过 Email + 网页横幅实现）
- ❌ 移动端 App（V1 做 responsive web）
- ❌ 支付处理/在线收款（V1 展示会员状态和贡献体系，但不接入 Stripe 等支付接口。会员身份由运营手动管理）
- ❌ 内容审核系统（V1 社群规模小，人工管理）
- ❌ 实时推送引擎（V1 用定时任务）

## 八、成功指标

### 核心指标（验证 V1 假设）

- **"我的页面"访问频率**：成员是否会回来看自己的参与记录？
- **投票/推荐/提案的活跃度**：网页端上线后，参与贡献的人数是否增加？
- **日/周活跃用户数**：是否真的"每天想来看看"？

### 人与人连接指标

- **感谢卡发送数**：活动后有多少人给 Host 或同场参与者发了感谢卡？
- **感谢卡公开比例**：有多少人选择公开感谢卡？（衡量社群信任度）
- **感谢卡回复率**：收到感谢卡的人有多少也回发了？（衡量连接是否双向）
- **话题讨论参与度**：每个话题的平均回复数
- **共同活动深度**：平均每对成员一起参加过几次活动？是否在增长？
- **跨活动类型连接**：成员是否在不同类型的活动中遇到同一个人？
- **成员页面互访频率**：看别人个人页面的频次

### Host 供给指标

- **月度活跃 Host 数**：每月至少发起 1 次活动的 Host 数量
- **新 Host 转化率**：收到引导后发起第一次活动的比例
- **Host 复做率**：Host 过 1 次的人中，3 个月内再次 Host 的比例
- **活动供给满足率**：Waitlist 比例越高说明供给越不足

### Agent 效果指标

- **推送打开率**：Email 推送的打开率
- **推送行动转化率**：打开后完成引导动作的比例
- **Agent 判断准确率**：人工审核期间，Agent 建议被采纳的比例
- **运营日报采纳率**：Agent 的运营建议被执行的比例

### 辅助指标

- 电影投票数对比 Notion 时期是否增长
- 活动报名从网页端完成的比例
- 新提案的提交频率
- 点赞和评论数量

## 九、未来方向（V2+）

记录但不开发：

- **统一发布入口**：用户在动态流里直接发内容，系统通过链接识别和引导将其结构化
- **更多推荐类型**：书、餐厅、音乐、菜谱、去处——复用电影池的模式

- **社群数据可视化**：整体的活动数量、参与人次、电影放映数等
- **Agent V2**：站内通知中心、A/B 测试推送文案、个性化推荐算法
- **Host 匿名反馈**：活动后参与者匿名评价体验
- **会员制集成**：会员费抵扣、贡献者免费等与 Host 激励体系打通

## 十、视觉设计风格

### 风格定位：深色 Lofi × 现代日式 × 社区工具

**一句话**：深色底做空间感，层级做立体感，lofi grain 做质感，单暖色做呼吸。这是一个社区工具，不是杂志也不是 SaaS 后台。

### 调性关键词

**是**：深色的、有层次的、克制的、有噪点的、工具感的、现代的
**不是**：古朴的、手工感的、暖白的、多彩的、卡通的、过度设计的

### 配色

全站只用**灰阶层级 + 一个暖色强调**，不引入第二个色相。

**背景层级**（通过色阶做立体感，不靠阴影）：
- 底层背景：`#0C0C0E`
- Surface 1（卡片）：`#141416`
- Surface 2（浮层/elevated）：`#1C1C1F`
- Surface 3（hover / active）：`#262629`
- 分割线：`#2A2A2F`

**唯一强调色**：暖金 `#D4A574`——用于日期标记、主要按钮、感谢卡背景光晕、关键操作。低饱和变体 `#D4A57433` 用于底色晕染。除此之外不引入其他色相。

**文字**：
- 主文字：`#E8E6E2`（暖白，不是纯白）
- 副文字：`#9A9894`
- 弱文字/占位：`#5A5854`

### 质感与纹理

**全局 Grain 覆盖**：Canvas 生成的噪点纹理覆盖全屏，混合模式 `screen`（深色底上噪点是亮的，像胶片颗粒感）。opacity 约 0.35-0.5。这是 lofi 感的核心来源。

**磨砂玻璃导航**：顶部 sticky 导航用 `backdrop-filter: blur(12px)` + 半透明背景 `${s1}cc`，增加深度层次。

**照片处理**：活动照片可叠加轻微降饱和 + grain，和整体质感融合。头像保持原色不处理。

**无阴影设计**：不用 box-shadow 做立体感。层次完全通过背景色阶区分——底层暗、卡片稍亮、浮层更亮。

### 排版

**布局**：
- 现代日式留白，内容不挤。卡片间距 12px
- 统一圆角 10-12px
- 移动端单列，桌面端最多双列
- Sticky 磨砂导航栏

**字体**：
- 中文：系统默认（PingFang SC / Hiragino Sans / Noto Sans CJK）
- 英文：系统 San Francisco / -apple-system
- 全站无衬线体，不用手写体或装饰字体（**例外**：感谢卡组件使用衬线字体模拟纸质质感）
- 正文 14px，行高 1.7，深色底上稍小字号更精致

**标题层级**：
- 大标题 17-22px semibold，副标题 15px regular
- 用字重区分层级，不堆字号差异

### 内容呈现

**人是主角**：
- 头像 28-34px，灰色底（`#262629`），浅色文字
- 头像组叠放展示参与者（-6px overlap），带 `+N` 计数
- 活动卡片优先展示 Host 头像 + 称号标签

**照片区域**：
- 使用渐变色块代替无照片占位（`s2` → `s3` 渐变 + accent 微光晕）
- 有照片时叠加底部渐变遮罩，标题压在图上
- 日期标签用磨砂 pill 浮在照片右上角

**电影海报**：
- 海报区域用深色渐变底，保持整体暗调统一
- 投票按钮独立，不叠在海报上

**标签（Tag）**：
- 背景 `#262629`，文字 `#9A9894`
- 小巧不抢眼，圆角 5px

**感谢卡**：
- 唯一使用暖纸色底（`#F5F0E8`）和衬线字体（Georgia / Noto Serif SC）的组件
- 模拟真实感谢卡质感：邮票组件、手写风排版、带齿孔边缘
- 公开感谢卡可带照片（场景渐变图），私密感谢卡带 🔒 标记
- 和深色主界面形成鲜明物理质感对比，强调"这是特别的"

### 交互感受

**轻快不沉重**：
- 页面切换淡入淡出（200-300ms）
- 点击反馈用 opacity 变化（0.7），不用 scale
- 加载状态用骨架屏，颜色接近 `s1`，不刺眼

**微动效克制使用**：
- 感谢卡发送成功 → 轻微 confetti 或心形动画（唯一的"花哨"动效）
- 投票成功 → 数字 +1 轻微弹跳
- 其他操作只用状态变化

**声音**：无。

### 与现有物料的衔接

- 海报噪点纹理 → 网页 grain overlay（从纸质噪点变为胶片颗粒，深色适配）
- 海报排版 → 活动详情页可参考类似布局
- 照片预设 → 网页端照片沿用同一套调色，叠加 grain 融入深色底

### 设计系统优先级

V1 先定义以下核心组件的视觉规范：
1. 动态流卡片（9 种类型：Agent 横幅、里程碑、活动、感谢卡、活动回顾、电影推荐、提案、种子计划、话题讨论）
2. 成员头像组件（头像 + 名字 + 称号标签 + 头像组叠放）
3. 按钮（主要 filled / 次要 outline / 文字链接 / 投票计数）
4. 电影海报卡片（渐变底 + 投票按钮）
5. 活动报名卡片（场景渐变图 + 社交上下文 + 两阶段邀请标识）
6. 感谢卡组件（暖纸色底 + 衬线字体 + 邮票 + 公开/私密标识）
7. 会员状态卡片（贡献者 / 被动会员两种视图）
8. Segment tab 切换器（圆角底 + 高亮当前项）
9. 空状态页面

其他组件在开发过程中迭代补充。
